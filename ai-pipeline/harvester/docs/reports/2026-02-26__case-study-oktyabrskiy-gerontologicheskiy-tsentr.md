# Case study: обогащение ОГБУ СО «Октябрьский геронтологический центр» (Костромская область)

**Дата:** 2026-02-26  
**Область:** Harvester — пайплайн обогащения организаций (поиск источника, верификация, harvest, мероприятия).  
**Источник истины:** [docs/Navigator_Core_Model_and_API.md](../../../docs/Navigator_Core_Model_and_API.md), [harvest-flows-a-b-c.md](../harvest-flows-a-b-c.md), [aggregators_guide.md](../aggregators_guide.md).

Организация использована как кейс для проверки полного цикла обогащения одной организации через скрипт `scripts/run_single_org_enrichment.py` (поиск/верификация → harvest → event discovery → опционально Core API).

---

## 1. Цель кейса

- Прогнать одну организацию через **существующую** инфраструктуру: EnrichmentPipeline, run_organization_harvest, EventDiscoverer, event_ingestion.
- Убедиться, что логирование кандидатов поиска и верификации даёт возможность разбирать сбои.
- Зафиксировать, на каких этапах пайплайн даёт сбой и почему (поиск, верификатор, harvest).

---

## 2. Организация

- **Название:** ОГБУ СО «Октябрьский геронтологический центр»
- **Регион:** Костромская область
- **Фактический сайт:** http://ogc-44.ru/ (вручную находится по запросу «Октябрьский геронтологический центр Костромская область»).
- **Тип:** государственное бюджетное учреждение соцобслуживания (пожилые, инвалиды).

---

## 3. Прогон 1: только поиск по названию и региону

**Команда:**  
`python -m scripts.run_single_org_enrichment "ОГБУ СО «Октябрьский геронтологический центр»" "Костромская область"`

**Провайдер:** YandexSearchProvider.

**Запрос к поиску:**  
`"ОГБУ СО «Октябрьский геронтологический центр»" Костромская область официальный сайт`

**Результат поиска (source_discovery_candidates):**

| № | URL | reachable | score |
|---|-----|-----------|-------|
| 1 | https://cpdirk.ru/... | True | 47.5 |
| 2 | https://src-zalarinskoe.ru/ | True | 32.5 |
| 3 | https://www.b2bsky.ru/companies/gerontologicheskiy_centr_... | True | 30.0 |
| 4 | https://socdeti.ru/ | True | 30.0 |
| 5 | https://vsepansionati.ru/kostroma/... | True | 25.0 |
| 6 | https://allpans.su/... | False | 22.5 |
| 7 | https://kostroma.spravkarf.ru/... | False | 22.5 |
| 8 | https://www.audit-it.ru/contragent/... | False | 22.5 |

**Итог:** Сайт организации **ogc-44.ru в топ-8 не попал**. В топе — другие сайты (cpdirk.ru, src-zalarinskoe.ru, b2bsky, socdeti, vsepansionati, справочники). Все 5 проверенных кандидатов получили при верификации `confidence=0.0`, `is_match=False` — корректно, это не сайт целевой организации.

**Важно:** Ручной поиск на **ya.ru/search** с тем же текстом запроса возвращает корректный URL (ogc-44.ru) первым. Мы же используем **Yandex Cloud Search API v2** (`searchapi.api.cloud.yandex.net`) — это **другой продукт**: другой индекс и другое ранжирование, не совпадающее с выдачей ya.ru. Плюс мы добавляем к запросу суффикс «официальный сайт», а на ya.ru пользователь мог искать без него; в публичном поиске поддерживается параметр региона `lr=213` (и др.), в нашем запросе к API регион не передаётся.

**Вывод по прогону 1:** При длинном точном названии («ОГБУ СО «Октябрьский геронтологический центр»») **наш** поиск (Yandex Cloud Search API v2) не выводит официальный сайт в топ. Для таких организаций поиск только по названию+региону недостаточен; имеет смысл передавать регион в API (если поддерживается) и/или пробовать укороченный запрос.

## 4. Прогон 2: задан URL, без skip-verify

**Команда:**  
`python -m scripts.run_single_org_enrichment "..." "Костромская область" --url "http://ogc-44.ru/"`

**Поведение:** Используется `enrich_broken_url`: поиск по фрагменту «ogc-44», затем верификация найденного URL.

**Результат верификации:**  
Один кандидат — `http://ogc-44.ru/`. SiteVerifier (лёгкий обход + LLM) вернул **confidence=0.0**, **is_match=False**.

**Итог:** Даже при передаче правильного URL пайплайн не принял его: верификатор отклонил страницу. Harvest не запускался.

**Вывод по прогону 2:** Для части реальных госсайтов верификатор даёт ложноотрицательный результат (страница не распознаётся как сайт организации). Требуется либо донастройка промпта/порогов, либо возможность обхода верификации для доверенных URL (например, `--skip-verify`).

## 6. Выводы и рекомендации (для использования как кейс)

1. **Поиск: API vs ya.ru**  
   Ручной запрос на ya.ru с тем же текстом возвращает ogc-44.ru первым; наш запрос идёт в **Yandex Cloud Search API v2** — другой продукт, другое ранжирование. Имеет смысл:
   - проверить в [документации Yandex Cloud Search API](https://yandex.cloud/en/docs/search-api/) поддержку параметра региона (аналог `lr` в ya.ru) и передавать регион (например, Костромская область) в запросе;
   - пробовать укороченный запрос («Октябрьский геронтологический центр Кострома») в рамках того же пайплайна или отдельного режима;
   - при наличии URL из агрегаторов (Silver Age, ФПГ, СО НКО) сразу передавать его в обогащение (enrich_broken_url / --url).

2. **Верификатор (SiteVerifier)**  
   Для http://ogc-44.ru/ верификатор дал 0.0 / not match. Целесообразно:
   - разобрать кейс (что вернул краул, какой фрагмент смотрит LLM);
   - при необходимости ослабить порог или доработать промпт;
   - для доверенных URL оставить возможность обхода верификации (как в скрипте с `--skip-verify`).

3. **ИНН и площадки**  
   В прогоне ИНН и venues не заполнились. На сайте они могут быть на страницах «Реквизиты»/«Контакты»; при необходимости:
   - расширить список обходимых путей в multi-page или дообход конкретных URL из уже загруженного списка;
   - при наличии ИНН из внешнего источника (Dadata, агрегатор) — передавать в пайплайн и в payload в Core.

4. **Мероприятия**  
   Страницы /news, /novosti, /afisha распознаны как страницы с событиями, но разбиение на отдельные события не дало кандидатов. Имеет смысл проверить разметку этих страниц и при необходимости доработать эвристики/селекторы в EventDiscoverer.

5. **Логирование**  
   Логи `source_discovery_candidates`, `enrichment_candidates`, `enrichment_verification` позволяют однозначно видеть, какие URL вернул поиск, какие пошли на верификацию и с каким результатом — кейс можно воспроизводить и использовать для отладки и доработки пайплайна.

## 7. Сравнение с ручным поиском на ya.ru

При запросе в браузере по адресу вида  
`https://ya.ru/search/?text=ОГБУ+СО+«Октябрьский+геронтологический+центр»,+Костромская+область&lr=213`  
пользователь видит корректный сайт (ogc-44.ru) первым. При программном фетче этого URL сервер ya.ru возвращает страницу **верификации** («Проверка браузера перед переходом на ya.ru»), а не результаты поиска, поэтому автоматически сравнить выдачу нельзя. Итог: мы используем Yandex Cloud Search API v2 — это не тот же движок, что ya.ru/search; различие в результатах ожидаемо.

## 9. Внесённые улучшения (после кейса)

По результатам разбора кейса внесены изменения:

1. **Регион в поиске Яндекса**  
   В пайплайн поиска добавлена передача параметра региона в Yandex Search API: при вызове `discover_sources(org_title, city=...)` регион (например, «Костромская область») маппится в коды Yandex (`region_name_to_yandex_lr`), и при использовании YandexSearchProvider в тело запроса добавляется `regionId`. Это должно улучшить ранжирование локальных организаций. См. `search/yandex_xml_provider.py`, `search/source_discoverer.py`, `search/provider.py`.

2. **Ложное срабатывание верификатора сайта**  
   Для разбора ложных отрицаний: в промпт верификатора добавлена подсказка по домену (коды регионов в домене — 44, 35 и т.д., сокращения ogc, kcson). При результате «не сайт организации» и confidence=0 логируется `site_verify_rejected` с превью текста страницы и reasoning LLM. См. `search/site_verifier.py`.

3. **Dadata в пайплайне обогащения**  
   В `EnrichmentPipeline` при отсутствии ИНН вызывается Dadata `suggest_party(название, регион)`. Полученные ИНН/ОГРН передаются в верификатор и после harvest мержатся в payload (ИНН, ОГРН, контакты, адрес как venue с fias_id/координатами из Dadata). Таким образом, легальная часть и контакты организации заполняются из реестра даже без сайта. См. `search/enrichment_pipeline.py` — `_dadata_suggest`, `_merge_dadata_into_harvest`.
