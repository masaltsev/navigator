## 1. Цель и методика

**Задача 1.10 из плана:** «Тест на 5 реальных КЦСОН через CLI — проверка цепочки Crawl4AI + DeepSeek → JSON».

**Методика:**

1. Из таблицы `sources` (kind = `org_website`) случайным образом отобраны 5 URL с разными типами организаций
2. Каждый URL обработан через Crawl4AI (fetch + markdown, без LLM-экстракции)
3. Полученный markdown передан в `OrganizationProcessor` → DeepSeek API → валидация Pydantic `OrganizationOutput`
4. Результаты **не записывались в БД** — только JSON-отчёт

**Почему не legacy CLI (`run_single_url.py`):** legacy-пайплайн использует `LLMExtractionStrategy` через LiteLLM/Crawl4AI, которая разбивает страницу на мелкие чанки и делает множество API-вызовов. При тестировании это приводило к таймаутам DeepSeek API (`[Errno 60] Operation timed out`). Новый пайплайн (`OrganizationProcessor` → OpenAI SDK напрямую) отправляет один запрос с полным system prompt и работает стабильно. Это **целевая архитектура** — legacy-пайплайн подлежит замене.

## 4. Анализ cache hit

| Вызов | Org | Input tokens | Cached | Cache % |
|-------|-----|-------------|--------|---------|
| 1 | КЦСОН | ~26K | 0 | 0% (cold start) |
| 2 | Неврология | 25 891 | 15 104 | **58.3%** |
| 3 | БФ «Образ жизни» | 24 609 | 15 104 | **61.4%** |
| 4 | Волонтёры 800 | 17 043 | 15 104 | **88.6%** |

System prompt (~15K tokens) кэшируется с первого вызова и переиспользуется. Чем меньше страница (→ меньше user message), тем выше % cache hit. При batch-обработке 5 000 организаций ожидается стабильный cache hit >90%.

## UPD 2026-02-24: Исправление критических проблем P1, P2, P4

### P1 — LLM путает коды org_type и ownership_type

**Статус: ИСПРАВЛЕНО** (два уровня защиты)

**Уровень 1 — промпт (`organization_prompt.py`, Правило 3):**

Добавлен явный блок с разъяснением разницы между справочниками:
- `organization_type_codes` → ЧТО это за организация (КЦСОН, поликлиника, НКО)
- `ownership_type_code` → КОМУ принадлежит (ГБУ, МБУ, АНО, фонд)
- Конкретный пример: «Код 161 (Благотворительный фонд) — это ownership_type, НЕ organization_type»

**Уровень 2 — автокоррекция (`OrganizationProcessor._validate_codes()`):**

Post-hoc перекрёстная проверка: если код из `organization_type_codes` найден в справочнике `ownership_types` — автоматически переносится в `ownership_type_code` (и наоборот). Логируется как info.

### P4 — LLM перефразирует title

**Статус: ЧАСТИЧНО ИСПРАВЛЕНО**

Добавлено Правило 5 в промпт: «`title` — ТОЧНОЕ официальное название организации, как указано на сайте. НЕ перефразируй, НЕ сокращай». Различать полное/краткое: полное → `title`, краткое → `short_title`.
