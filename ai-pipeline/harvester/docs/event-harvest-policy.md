# Бизнес-логика: когда запускать сбор мероприятий (events)

**Дата:** 2026-02-25  
**Связанные документы:** [harvest-flows-a-b-c.md](harvest-flows-a-b-c.md) (потоки А–В по организациям и место событий), [event_ingestion_pipeline.md](event_ingestion_pipeline.md) (как обрабатываются мероприятия после решения «запускать»).

Отдельный проход по мероприятиям (`harvest_events`, POST /harvest/events) сегодня вызывается вручную. Ниже зафиксированы правила: когда его запускать, когда не запускать, чтобы не дублировать работу и не пропускать агрегаторы мероприятий.

---

## Правило в одном предложении

**Отдельный проход по мероприятиям запускаем только для источников-агрегаторов мероприятий (афиши, разделы событий Silver Age и т.п.). Для обычного сайта организации не запускаем отдельный проход, если мероприятия уже обнаружены на странице организации при обычном обходе.**

---

## Два случая

### 1. Запускать отдельный проход по событиям — **да**

Когда источник по смыслу является **агрегатором мероприятий**:

- Афиша мероприятий «Серебряное долголетие» (отдельный сайт/раздел).
- Раздел мероприятий на платформе Silver Age (мероприятия как основной контент).
- Любой источник, у которого основная цель — список событий, а не описание организации.

Для таких источников:

- Имеет смысл вызывать `harvest_events` (или аналог) **отдельно** от обхода организации.
- В Core можно завести отдельный `kind` источника (например `event_aggregator` или `platform_silverage_events`) и по нему оркестратор решает, что по этому источнику нужно запускать сбор событий.

### 2. Отдельный проход по событиям — **нет**

Когда мероприятия **уже обнаружены при обычном обходе сайта организации**:

- Обычный проход: `crawl_and_enrich` → MultiPageCrawler (или в будущем расширенный crawl) уже заходит на /news, /afisha, /meropriyatiya и т.п.
- В том же проходе выполняется event discovery по уже скачанному контенту (например, через `EventDiscoverer.discover_from_cached_markdown` или добавление event-страниц в общий краул).
- События классифицируются и отправляются в Core в рамках этого же прогона.

В этом случае **отдельно** вызывать `harvest_events` по тому же URL не нужно:

- Избегаем дублирования краула и дублирования событий.
- Один проход = одна организация + её мероприятия.

---

## Текущее состояние в коде

- **MultiPageCrawler** специально **исключает** из обхода страницы типа /news, /afisha, /press (чтобы не раздувать контекст для LLM по организации). То есть сейчас мероприятия при обходе организации **не** собираются.
- **EventDiscoverer** умеет:
  - `discover_events(base_url, main_page_markdown=None)` — сам краулит /news, /afisha и т.д.;
  - `discover_from_cached_markdown(base_url, event_page_markdowns)` — разбор уже скачанных event-страниц (вариант «мероприятия в том же проходе»).
- **harvest_events** (Celery + POST /harvest/events) всегда делает отдельный краул event-страниц; связи с «этот источник уже обошли с событиями» или «это агрегатор событий» в оркестраторе пока нет.
- **Обработка** найденных мероприятий (любым путём) идёт через [универсальный пайплайн приёма мероприятий](event_ingestion_pipeline.md): адаптер → RawEventInput → `run_event_ingestion_pipeline` → Core API.

Итого: явной бизнес-логики «когда запускать сбор событий» в оркестраторе пока нет; политика реализована в коде в `harvest.event_harvest_policy`.

---

## Рекомендуемые шаги

1. **Политика в коде (Harvester)**  
   Модуль/функция, по которой оркестратор решает, нужно ли для данного источника запускать отдельный event harvest:
   - Вход: `source_kind`, `base_url`, опционально флаг `events_already_harvested_in_org_run`.
   - Выход: `True` — запускать отдельный проход по событиям, `False` — не запускать.
   - См. `harvest/event_harvest_policy.py`.

2. **Оркестратор (Laravel / Scheduler)**  
   При формировании очереди обхода:
   - Для источников с `kind = org_website` (и аналогами): по умолчанию **не** вызывать POST /harvest/events для того же URL после crawl_and_enrich (пока не реализовано «события в том же проходе» — можно вообще не слать event harvest для обычных сайтов).
   - Для источников-агрегаторов мероприятий (новый `kind` или белый список URL): вызывать POST /harvest/events (или эквивалент).

3. **Опционально: события в том же проходе, что и организация**  
   В `run_organization_harvest` (или после него в tasks):
   - Расширить краул так, чтобы получать и event-страницы (или передавать в EventDiscoverer уже скачанный markdown).
   - Вызывать EventDiscoverer (например, `discover_from_cached_markdown`), затем классификацию событий и import/event в Core.
   - Помечать, что для данного source в этом run события уже обработаны (чтобы оркестратор не ставил в очередь отдельный harvest_events).

4. **Core**  
   При появлении агрегаторов мероприятий — ввести `kind` (например `event_aggregator`, `platform_silverage_events`) и в API due sources отдавать его, чтобы оркестратор мог различать «источник организации» и «источник мероприятий».

---

## Итоговая матрица

| Ситуация | Отдельный проход harvest_events? |
|----------|----------------------------------|
| Источник — агрегатор мероприятий (афиша, Silver Age события) | **Да** |
| Обычный org_website, мероприятия уже найдены в том же проходе (org crawl) | **Нет** |
| Обычный org_website, мероприятия в проходе не искали | По текущей реализации — только вручную; автоматически не запускать, пока не появится «события в том же проходе» или явная настройка на источник |
