# **Проектирование первого этапа AI Pipeline и Harvester для платформы «Навигатор здорового долголетия»**

Разработка масштабируемого конвейера сбора и обогащения данных (AI Pipeline) для социальных, медицинских и волонтерских сервисов требует фундаментального пересмотра традиционных методов агрегации информации. Исторически сложившиеся подходы к веб\-скрейпингу, опирающиеся на жесткие селекторы каскадных таблиц стилей (CSS) и регулярные выражения, демонстрируют критическую уязвимость перед постоянной эволюцией структур веб\-страниц, что делает их абсолютно непригодными для работы с тысячами разнородных ресурсов.1 В контексте проекта «Навигатор здорового долголетия» первый этап развертывания интеллектуального сборщика (Harvester) инициируется с анализа уже существующих в базе данных организаций, для которых в таблице источников зафиксированы адреса официальных веб\-сайтов под идентификатором типа ресурса. Внедрение больших языковых моделей (LLM) и концепции агентного искусственного интеллекта (Agentic AI) позволяет кардинально трансформировать этот процесс: интеллектуальный агент не просто извлекает сырой гипертекст, но семантически интерпретирует контент, классифицирует его и выравнивает в строгом соответствии с внутренней реляционной моделью данных проекта.2

Настоящий исследовательский и архитектурный отчет представляет исчерпывающую методологию развертывания интеллектуального пайплайна. Особое внимание в документе уделяется созданию полиморфной системы инструкций, адаптивному графовому обходу веб\-ресурсов, прецизионной темпоральной экстракции мероприятий для целевой аудитории старшего возраста, а также внедрению бескомпромиссных протоколов безопасности для защиты верифицированных массивов данных.

## **Интеграция с базовой архитектурой данных и механизм строгого семантического ограничения**

Фундаментальным требованием к проектируемому конвейеру является полная синхронизация результатов работы языковых моделей с документом, описывающим базовую архитектуру ядра системы и ее программных интерфейсов. Языковая модель, предоставленная самой себе, склонна к генерации правдоподобных, но технически некорректных или несуществующих категорий — явлению, известному как галлюцинации.5 Для предотвращения терминологического загрязнения базы данных «Навигатор здорового долголетия» вводится концепция принудительного семантического ограничения (Schema-Constrained Information Extraction).6

Суть данного подхода заключается в том, что нейросеть никогда не должна получать свободу в формулировании названий услуг, тематик или категорий специалистов для прямой записи. Архитектура Harvester включает промежуточный слой (Middleware), который динамически загружает актуальные словари-сидеры из базы данных проекта и интегрирует их непосредственно в контекст запроса (Prompt) или использует на уровне параметров API модели, таких как механизмы структурированного вывода (Structured Outputs).1

Процесс обработки извлеченной информации строго формализуется. При обнаружении на веб\-сайте описания определенной деятельности, языковая модель вычисляет косинусное сходство или использует логический вывод для сопоставления найденного текста с закрытым перечнем допустимых терминов из сидеров.2 Если модель определяет, что обнаруженная деятельность соответствует утвержденному термину «Оздоровительная физкультура для пожилых», в итоговый пакет данных вписывается исключительно этот верифицированный идентификатор.3 В случаях, когда обнаруженная на сайте услуга не имеет аналогов в словарях-сидерах, модель инструктируется игнорировать данный элемент при записи в основные таблицы, помещая его в буфер.5

**Анализ и предложения новых терминов:** Важным дополнением к данному процессу является механизм генерации предложений. Искусственному агенту ставится задача не только категоризировать услуги по имеющимся сидерам, но и выявлять закономерности в неклассифицированных активностях. По мере накопления данных, агент должен формировать аналитические отчеты с предложениями новых терминов для справочников. Эти предложения ни в коем случае не вносятся в базу данных самостоятельно, а складываются в отдельную очередь для рассмотрения администраторами, что обеспечивает контролируемое развитие таксономии проекта.26

## **Эволюция системы инструкций: Переход к полиморфному промптингу**

Текущая база данных «Навигатор здорового долголетия» характеризуется высочайшей степенью гетерогенности. В ней сосуществуют государственные больницы, региональные поликлиники, комплексные центры социального обслуживания населения (КЦСОН), центры общения старшего поколения Социального фонда России (ЦОСП СФР), локальные досуговые клубы по интересам и стихийные инициативные волонтерские группы.9

Применение единого набора инструкций к столь разнообразным субъектам неизбежно приводит к искажению данных. Для решения проблемы вводится архитектура полиморфного промптинга (Polymorphic Prompt Engineering), базирующаяся на принципах ансамблирования и композиции инструкций.10 Система самостоятельно классифицирует тип изучаемой организации и динамически собирает специализированный контекстный запрос (Contextual Prompt).

**Адаптивное «мышление» агента при классификации:** Стоит особо подчеркнуть, что приведенный базовый перечень типов организаций не должен быть закрытым. Применение исключительно жесткого списка лишит агента гибкости. Агенту предоставляется возможность «самостоятельного мышления»: если он анализирует веб\-сайт, структура и смысловое наполнение которого не соответствуют ни одному из заранее заданных профилей (например, это новый формат эко-инициативы для пенсионеров), агент использует Zero-Shot или Few-Shot классификацию для вывода новой категории из контекста.31 В этом случае он помечает организацию тегом «Другое / Требует классификации» и генерирует гипотезу о новом типе учреждения для последующего обогащения базового справочника администратором.

Подобный композитный подход гарантирует, что интеллектуальный агент будет применять релевантную оптику при анализе текста.11 Кроме того, многосоставной промпт включает в себя обязательный блок правил формата, который принуждает нейросеть возвращать обогащенные данные в строго типизированной структуре.7

## **Интеллектуальная навигация и стратегия графового обхода источников**

Стратегия обхода строится на гибридном применении эвристического поиска и семантического анализа гиперссылок (Agentic Web Scraping).4 Когда Harvester загружает веб\-страницу, языковая модель изучает все доступные навигационные узлы и вычисляет вероятность нахождения целевой информации по каждому из направлений.5 Агент инструктируется отдавать абсолютный приоритет ссылкам, семантически связанным с деятельностью, услугами, контактами и мероприятиями.

Для медицинских и социальных организаций (КЦСОН), агент использует заранее известные паттерны ожидаемой регламентированной разметки, активируя скрипт целенаправленного поиска нужных разделов.13 Для предотвращения бесконечного циклирования внедряются строгие механизмы контроля обхода: бюджет глубины (Depth Limit) и реестр посещенных адресов.17 Дополнительно внедряются фильтры шумовых страниц (политики обработки данных, тендеры и т.д.). В процессе обхода агент также аккумулирует ссылки на профили организации в социальных сетях, подготавливая базу для их будущего парсинга.18

## **Темпоральная экстракция и прецизионная верификация актуальных мероприятий**

Одной из самых амбициозных задач конвейера является автоматизированный поиск, извлечение и валидация мероприятий для лиц старшего возраста. Контент подобного рода обладает высокой временной волатильностью, следовательно, интеллектуальный агент должен обладать развитыми механизмами темпорального логического вывода (Temporal Reasoning).19

Основываясь на анализе деятельности центров, фокус поиска смещается в сторону лекций, мастер-классов, музыкальных вечеров и программ активного долголетия.9 Тексты на веб\-сайтах редко содержат даты в удобном для машин формате, используя нечеткие выражения.21 Для преодоления этого барьера внедряется двухступенчатая архитектура экстракции.19

На первом этапе языковая модель осуществляет распознавание сущностей. В контекст (Prompt) модели передаются текущая дата (current\_date) и дата публикации (publication\_date), опираясь на которые модель вычисляет точные значения для относительных выражений.22 На втором этапе извлеченные данные трансформируются в стандартизированную модель временного диапазона с атрибутами valid\_at и invalid\_at.2222

Алгоритм сервера сравнивает valid\_at с текущим временем, безвозвратно отсеивая прошедшие события.23 Для циклических мероприятий агент формирует структурированные правила повторения (RRule), позволяя программно генерировать будущие даты.24

## **Комплексное обогащение профилей и трансформация контента**

Собрав информацию, интеллектуальный конвейер агрегирует данные для их интеграции в систему «Навигатор здорового долголетия».

Ключевой задачей является обновление дескрипшна организации: языковая модель получает инструкцию выполнить смысловую компрессию сложного бюрократического контента, сгенерировав емкое и понятное описание для целевой аудитории.

Параллельно происходит обогащение реляционных связей:

* **Синхронизация сервисов и тематик:** Все верифицированные услуги добавляются в профиль организации.  
* **Интеграция специалистов:** Извлекаются и структурируются профили сотрудников.12  
* **Добавление актуальных мероприятий:** Валидированные события формируют динамическую афишу активностей.  
* **Расширение списка источников:** Найденные ссылки на социальные сети формализуются и записываются в таблицу sources.18

## **Протоколы безопасности данных и защита верифицированных полей**

Операции прямого обновления (Direct UPDATE) результатов работы ИИ в критические таблицы строго запрещены во избежание галлюцинаций модели и затирания верифицированной информации (например, из сервиса Dadata).5 Архитектура конвейера строится на базе промежуточных буферных зон и концепции изоляции полей.26

Процесс обновления состоит из следующих этапов:

1. **Теневое сохранение (Staging):** Структурированные данные записываются во временные таблицы (staging\_organizations\_updates). Основная база остается нетронутой.  
2. **Генерация дифференциалов (Diff Analysis):** Вычислительный модуль сравнивает текущее состояние профиля с пакетом обновлений.  
3. **Применение политик неизменности (Immutability Rules):** Жесткий программный фильтр исключает из обновления любые поля, имеющие маркер верификации (verified\_by\_dadata \= true), такие как юридическое наименование, ИНН, ОГРН, юридический адрес.26  
4. **Управление происхождением данных (Data Lineage):** В базу записывается ID сеанса, URL-адрес источника, временная метка и метрика уверенности модели для возможности отката изменений.2926

При конфликтах валидации пакет обновлений маршрутизируется в очередь для ручного аудита (Human-in-the-Loop).26

## **Рекомендуемый технологический стек конвейера**

Учитывая, что текущий бэкенд платформы реализован на PHP-фреймворке Laravel в связке с реляционной СУБД PostgreSQL, архитектуру сбора и обработки данных целесообразно вынести в отдельный асинхронный микросервис (или набор воркеров).

* **Ядро Harvester (Python):** Python является безусловным стандартом в сфере ИИ-разработки. Для реализации пайплайна рекомендуется использовать современные библиотеки агентного веб\-скрейпинга, такие как ScrapeGraphAI 3 (позволяет извлекать данные с помощью LLM с привязкой к навигационным графам и Pydantic-схемам), либо мощные инструменты вроде Firecrawl 32 и Crawl4AI 8, которые оптимально парсят и очищают сложный веб\-контент для подачи в языковые модели.  
* **Очереди и асинхронное взаимодействие:** Интеграция между ядром на Laravel и Python-пайплайном должна быть построена на брокере сообщений (Redis или RabbitMQ).33 Laravel инициирует задачу (передавая URL, тип организации и базовые словари), помещая её в очередь. Python-воркер забирает задачу, выполняет обход, обогащает данные и отправляет ответный payload через webhook или отдельную очередь результатов.  
* **Промежуточное хранение (Staging базы):** Обогащенные JSON-объекты сохраняются в PostgreSQL в специализированные таблицы staging\_organizations, где происходит валидация (проверка Diff'ов и сверка с флагами Dadata) перед выполнением безопасного слияния (UPSERT) в боевые реляционные таблицы.26
