## 1. Что сделано со старта проекта

### 1.1. Запуск бэкенда, экспорт БД из WordPress и рефакторинг структуры данных

**Зачем:** Старый каталог работал на WordPress и плагине HivePress. На объёме около 6 000 карточек уже проявлялись проблемы: медленные запросы, некорректные результаты при фильтрации по карте и категориям. Конверсия посетителей в целевые действия (звонок, форма) — около 3%. Для роста базы и возможности запуска дополнительных приложений (VK Mini Apps, iOS/Android) нужен был отдельный «движок» — независимый бэкенд с нормальной структурой данных.

**Что сделано:**

- **Новый бэкенд (Navigator Core)** на Laravel: API для организаций, мероприятий, площадок, справочников. Данные хранятся в реляционной структуре (PostgreSQL), а не в «плоских» метаданных WordPress.
- **Экспорт и миграция из WordPress:** все листинги (организации) перенесены в новую базу. Итог: **~6 048 организаций**, **~5 508 площадок**, **12 статей**, справочники заполнены и связаны с организациями. Координаты исправлены (устранён исторический баг с перепутанными широтой/долготой).
- **Рефакторинг структуры данных и онтологии:** вместо одной «карточки со всем подряд» — разделение на организации, площадки (адреса), организаторов. Не менее важно: переход от логики «категории проблем» к логике **жизненных ситуаций** (тематические категории в пользовательском языке). Справочники выстроены вокруг трёх доменов: **Здоровье и уход**, **Быт и социальная поддержка**, **Активная жизнь** — в соответствии с международными практиками (ВОЗ, Индекс активного долголетия ООН, СДУ РФ). Пользователь ищет не «диагноз», а ситуацию («хочу быть полезным», «одиночество», «нужна помощь с жильём»); ИИ и каталог работают в этой же логике. Подробнее см. [исследование по обновлению онтологии](./navigator_ontology_update.md).
- **Техническая основа:** быстрые запросы по карте и фильтрам, готовность к подключению новых витрин (сайт, VK, OK, мобильные приложения).

**Итог по этапу:** Каталог вынесен из WordPress в отдельное ядро. Данные приведены к единой модели, готовой к масштабированию и к приёму результатов от ИИ-пайплайна.

### 1.2. Первая фаза запуска ИИ-пайплайна и Harvester

**Зачем:** Ручное наполнение и обновление тысяч карточек не масштабируется. Нужен «виртуальный ассистент»: автоматический обход сайтов организаций и реестров, проверка релевантности (работа с пожилыми), классификация по справочникам и отправка только проверенных данных в базу.

**Что сделано:**

- **Harvester (сборщик данных):** Python-сервис, который:
  - заходит на сайт организации (главная + ключевые подстраницы: контакты, о нас, услуги, реквизиты);
  - извлекает текст и структуру страниц (в т.ч. через браузерную автоматизацию и при необходимости через облачный сервис для «сложных» сайтов);
  - передаёт контент в нейросеть (DeepSeek), которая определяет: подходит ли организация под критерии «Навигатора», какие услуги и категории указывать, контакты, адреса;
  - обогащает адреса через Dadata (нормализация, координаты, ФИАС);
  - отправляет результат в бэкенд по внутреннему API (организации и при необходимости мероприятия).
- **Умная маршрутизация решений:** ИИ не просто «да/нет», а три исхода: «одобрено» (высокая уверенность), «на модерацию» (сомнения), «отклонено». Порог уверенности и флаг «работа с пожилыми» задают, попадёт ли карточка сразу в каталог или в очередь на проверку человеком.
- **События (мероприятия):** реализовано обнаружение страниц с анонсами (новости, афиша, события), извлечение мероприятий из текста и классификация через отдельную модель. Результаты можно отправлять в Core (импорт событий).
- **Пакетная обработка и мониторинг:** задачи можно ставить пачками (десятки URL), логи и метрики (успех/ошибки, уверенность ИИ, стоимость токенов) собираются для анализа.

**Итог по этапу:** Первая фаза ИИ-пайплайна запущена: один URL или пакет URL проходят цепочку «сайт → сбор → ИИ → обогащение → API Core». Инструмент готов к первому полному проходу по списку сайтов организаций и к пилоту по сбору мероприятий.

### 1.3. Исследования, тесты и эксперименты

**Исследования и проектирование** (вели разработку; суть без привязки к конкретным артефактам):

- **Целевая архитектура и модель данных** — доменная модель ядра, публичный и внутренний API, жизненный цикл сущностей (черновик → модерация → одобрено/отклонено), подсистема источников и Harvester, стратегия миграции из старой системы.
- **Проектирование миграции из WordPress** — пошаговый маппинг полей старой БД в новую структуру, двухэтапная стратегия (сначала перенос «как есть», затем обогащение через ИИ), исправление координат и перенос таксономий в справочники.
- **Концепция ИИ-пайплайна** — жёсткое ограничение вывода нейросети только кодами из справочников (без «выдуманных» категорий), полиморфные инструкции под типы организаций (медицина, соцзащита, активное долголетие), извлечение дат и событий из текста, правила безопасности (не перезаписывать верифицированные человеком или Dadata поля).
- **Спецификация и план Harvester** — спринты разработки, стратегии обхода сайтов (одна страница → несколько страниц → CSS-шаблоны), интеграция с ядром, пакетная обработка и контейнеризация.
- **Рефакторинг онтологии** — переход от «категорий проблем» к жизненным ситуациям и трём доменам (Здоровье и уход, Быт и соцподдержка, Активная жизнь), расширение справочников для ИИ (услуги, типы организаций, профили специалистов) в логике Positive Aging; обоснование по стандартам ВОЗ, ООН, СДУ РФ.
- **Промпты и инструкции для ИИ** — формулировки, по которым нейросеть классифицирует организации и мероприятия, проверяет релевантность «работа с пожилыми» и возвращает структурированный ответ без лишних категорий.

**Тесты и эксперименты** (что проверяли на практике):

- **Прогон миграции из WordPress** — полный перенос датасета (6 217 листингов), проверка счётчиков и связей, выявление и исправление дубликатов и координат.
- **Проверка API ядра** — списки и карточки организаций, фильтры по жизненным ситуациям, типам, услугам; события; внутренний API приёма данных от Harvester (организации и мероприятия).
- **Пилот обхода одного сайта** — цепочка «сайт → сбор текста → ИИ → обогащение адресов → отправка в ядро» на 5 реальных URL (КЦСОН, медучреждение, НКО, культура, волонтёры): 4 из 5 успешно; для «сложных» сайтов (SPA) добавлен облачный fallback.
- **Мультистраничный обход** — обход главной и ключевых подстраниц (контакты, о нас, услуги, реквизиты) и слияние контента в один вход для ИИ; проверка на типовых структурах сайтов.
- **Пакетная обработка** — постановка десятков URL в очередь, распределённая обработка, сбор метрик (успех/ошибки, уверенность ИИ, стоимость токенов).
- **Сбор мероприятий с сайтов** — автоматическое обнаружение страниц с анонсами (новости, афиша), извлечение событий из текста, классификация через ИИ и отправка в ядро.
- **Готовность к продакшену** — структурированные логи, централизованные метрики, валидация URL перед обходом, тесты (сотни сценариев в модуле Harvester).

## 2. Как это работает (упрощённо)

1. **Пользователь** (пока — текущий сайт на WP) ищет организацию или услугу. В перспективе к одному Core подключатся обновлённый сайт, мини-приложения VK/OK, мобильные приложения.
2. **Core (Laravel)** отдаёт данные из своей БД: организации, площадки, мероприятия, справочники (жизненные ситуации, услуги, типы организаций и т.д.). Фильтры и карта работают по нормализованным полям и геометрии.
3. **Наполнение и обновление каталога:** Core хранит список источников (сайты организаций, реестры). Планировщик или админ инициирует «обход». Harvester обходит сайты, ИИ классифицирует и проверяет релевантность, при необходимости обогащает адреса через Dadata и отправляет результат в Core. Core решает: сразу опубликовать, отправить на модерацию или отклонить.
4. **Мероприятия:** Harvester находит на сайтах страницы с анонсами, извлекает события, классифицирует их и может отправлять в Core для отображения в афише.

Таким образом, бизнес-процесс меняется: ручное обновление карточек дополняется (и со временем может замещаться) автоматическим сбором и верификацией через ИИ с контролем качества (уверенность, модерация).

## 3. Тесты и проверки

- **Миграция WP:** прогон на полном датасете (6 217 листингов), проверка счётчиков организаций/площадок/связей, исправление дубликатов и координат.
- **API бэкенда:** проверены эндпоинты списков и деталей организаций, фильтры (жизненные ситуации, типы, услуги), события; внутренний API импорта (organizer, event) и контракт с Harvester.
- **Harvester:** тесты на 5 реальных URL (КЦСОН, медицина, НКО, культура, волонтёры): 4/5 успешно; один отказ — SPA-сайт, для таких добавлен облачный fallback. Пакетная обработка (Celery), сбор мероприятий, логирование и метрики покрыты тестами (сотни тестов в модуле).

## 4. Изменения в концепции и новые возможности

- **От «категорий проблем» к «жизненным ситуациям»:** онтология каталога переосмыслена. Раньше фокус был на медицинских и социальных «проблемах». Теперь справочники построены как **жизненные ситуации** — язык пользователя («одиночество», «хочу передать опыт», «цифровая изоляция», «адаптация жилья») и три домена: Здоровье и уход, Быт и социальная поддержка, Активная жизнь. ИИ классифицирует и карточки, и запросы в этой логике — так реализуется «ответ вместо поиска»: человек описывает ситуацию, система подбирает организации и материалы.
- **От «каталога» к «ответу вместо поиска»:** целевая концепция — не просто фильтр по категориям, а связывание запроса пользователя с конкретными организациями и материалами. Для этого заложены нормализованные справочники (уже в логике жизненных ситуаций), события и в перспективе статьи с привязкой к тематикам и услугам.
- **Организаторы не только юрлица:** в модели заложены полиморфные организаторы (организация, инициативная группа, физлицо). Сейчас в проде в основном организации из WP; механизм готов к добавлению инициатив без «поддельных» юрлиц.
- **События как объект первого класса:** мероприятия с датами и повторениями (RRule), материализованные экземпляры — основа для афиши и фильтров «что происходит рядом».
- **ИИ как часть процесса, а не разовый скрипт:** классификация и проверка «работа с пожилыми» встроены в пайплайн; уверенность ИИ и статусы (approved / pending_review / rejected) управляют видимостью без потери контроля.

**Вау-эффект:** За один прогон по URL можно получить обновлённую структурированную карточку (в т.ч. контакты, адреса, категории) и список мероприятий с сайта, с минимальным ручным трудом. При масштабировании это даёт возможность поддерживать актуальность тысяч карточек и растущую афишу событий.

## 6. Перспективы и следующие шаги

- **Ближайшее:** первый полный проход Harvester по списку сайтов организаций (тысячи URL), разбор метрик и качества; при необходимости — доводка порогов и модерации. Параллельно — проверка/запуск материализации событий (event_instances) и публичного API событий.
- **Кампания ИРИ (март 2026):** текущий сайт оптимизируется под лидогенерацию; новый Core и обновлённый каталог — для перевода прогретой аудитории после релиза.
- **Дальше:** подключение новых витрин (обновлённый сайт, VK/OK мини-приложения), расширение источников (агрегаторы, реестры), развитие логики «ответ вместо поиска» (статьи + рекомендации).

Итог: фундамент заложен — каталог вынесен из WordPress, данные приведены к целевой модели, первый контур ИИ-пайплайна и сбора мероприятий работает. Оставшийся объём — доведение оркестрации, событий и публичных API до целевой спецификации и масштабирование обхода источников.
