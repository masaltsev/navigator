# **Архитектурная доменная модель и проектирование API для сервиса Navigator Core**

## **Краткое экспертное резюме: Цели и предпосылки рефакторинга**

Представленный ниже архитектурный проект решает критические проблемы текущей инфраструктуры платформы и закладывает масштабируемый фундамент для развития сервиса. Необходимость перехода к новому независимому ядру продиктована не только техническими ограничениями, но и масштабными продуктовыми целями:

1. **Преодоление жестких ограничений производительности:** Текущая монолитная архитектура на базе WordPress и плагина HivePress исчерпала свои возможности. Недавние UX-тесты показали, что сложные запросы — в частности, фильтрация по геопозиции на карте в связке с категориями проблем и услуг — работают недопустимо медленно и выдают некорректные результаты уже на объеме всего в 6 000 карточек организаций. С учетом стратегии по кратному увеличению базы данных, сохранение старой структуры неизбежно приведет к зависаниям и коллапсу системы при масштабировании.  
2. **Повышение пользовательской конверсии:** По данным Яндекс.Метрики, текущая конверсия посетителей сайта в целевые действия (клик по номеру телефона, заполнение и отправка форм, переход в социальные сети) зафиксирована на уровне всего 3%. Переход от устаревшего «глухого» каталожного поиска к парадигме «Ответ вместо поиска» (когда система связывает проблему пользователя с конкретными организациями и обучающими статьями) позволит радикально улучшить пользовательский опыт и кратно повысить эту метрику.  
3. **Автоматизация сбора и проверки данных (AI-пайплайн)**: Сейчас наполнение каталога, пусть и работает с помощью ИИ, требует много ручного труда – по сбору первичного датасета, мониторингу изменений в источниках данных. Новая архитектура включает умный конвейер, который работает как виртуальный ассистент. Он автоматически регулярно обходит сайты реестров и социальных сетей, собирает сырую информацию и с помощью нейросетей проверяет, действительно ли организация помогает пожилым людям. Искусственный интеллект сам классифицирует услуги по справочникам и только проверенные, качественные данные отправляет в базу. Это позволит поддерживать актуальность десятков тысяч карточек без раздувания штата сотрудников.  
4. **Омниканальность и гибкость интерфейсов (API-first)**: Переход на независимый бэкенд отвязывает нас от визуальной части сайта. Это означает, что к одному мощному "движку" базы данных можно будет легко подключать любые внешние витрины. На первом этапе мы запустим обновленный веб\-сайт и мини-приложение ВКонтакте (VK Mini App), а в будущем сможем без переделки ядра выпустить полноценные мобильные приложения для iOS/Android или интегрироваться с партнерскими сервисами.  
5. Адаптация к федеральному трафику (Кампания ИРИ): В марте 2026 года запланирован старт масштабной рекламной кампании при поддержке Института развития интернета (ИРИ) с прогнозируемым охватом 50–70 млн человек. Учитывая цикл разработки, полноценный запуск Navigator Core до этого срока невозможен. В связи с этим принята гибридная стратегия: на время кампании текущий сайт будет максимально оптимизирован для задач лидогенерации. Мы лучше структурируем основные контентные разделы и будем собирать контакты пользователей в «лист ожидания», чтобы после релиза новой платформы бесшовно перевести туда готовую, прогретую базу пользователей.

## **Концептуальные основы доменной модели**

Проект «Навигатор здорового долголетия» находится на этапе стратегического транзита от концептуальной бета-версии к полномасштабной основной версии, призванной стать центральным элементом экосистемы «серебряной экономики». Исторически сложившаяся архитектура, базирующаяся на системе управления контентом WordPress и плагине HivePress, исчерпала свой потенциал масштабируемости. Модель данных Entity-Attribute-Value (EAV), используемая в WordPress для хранения метаданных (postmeta), создает непреодолимые узкие места при выполнении сложных геопространственных запросов, фильтрации по пересекающимся временным интервалам и агрегации полиморфных сущностей. В связи с этим возникает критическая необходимость в проектировании независимого высокопроизводительного backend-сервиса Navigator Core на базе фреймворка Laravel, который будет функционировать в симбиозе с выделенным AI-пайплайном.

Новая архитектура фундаментально меняет парадигму взаимодействия пользователя с платформой, осуществляя переход от традиционного директивного поиска к модели «Answer Instead of Search» (Ответ вместо поиска). Для поддержки этой функциональности доменная модель должна обеспечивать строгую нормализацию адресных данных через интеграцию с сервисом Dadata, сложную маршрутизацию статусов на основе метрик уверенности искусственного интеллекта (AI Confidence Scores), а также поддержку сложных паттернов повторяющихся событий (Recurrence Rules), характерных для регулярных активностей пожилых людей. Настоящий отчет представляет собой исчерпывающее проектирование целевой доменной модели, контрактов прикладного программного интерфейса (API) и стратегий реализации, гарантирующих высокую доступность, строгую типизацию и готовность к бесшовной интеграции с внешними платформами, такими как мини-приложения ВКонтакте и Одноклассников.

Проектирование целевой доменной модели начинается с деконструкции монолитного представления организации. В предыдущей итерации любая активность или точка на карте жестко привязывалась к сущности hp\_listing, которая де\-факто отождествлялась с юридическим лицом. Однако анализ реальных социологических и административных паттернов в сфере серебряного добровольчества выявляет дихотомию между формальными организациями и неформальными институтами. Формальные организации обладают четкой иерархией, юридическим статусом, ИНН и регламентированными процедурами. В противовес им неформальные организации (например, инициативные группы пожилых людей) формируются на базе социальных сетей, общих интересов и не имеют строгой юридической оболочки, однако генерируют значительный объем социальной активности и мероприятий.

Чтобы органично инкапсулировать обе парадигмы без искусственного раздувания базы данных фиктивными юридическими лицами, доменная модель Navigator Core внедряет паттерн полиморфных связей. Этот подход позволяет родительской сущности (например, Мероприятию) ссылаться на абстрактного «Организатора», который на физическом уровне базы данных может разрешаться либо в профиль зарегистрированного фонда, либо в карточку инициативной группы, обеспечивая тем самым высочайшую архитектурную гибкость.

**Примечание о стандартных полях Laravel:** Все основные сущности (organizations, events, venues, articles, sources, event_categories) используют стандартные поля Laravel `timestamps` (created_at, updated_at) для отслеживания времени создания и обновления записей. Большинство сущностей также используют `softDeletes` (deleted_at) для логического удаления, что позволяет восстанавливать записи и сохранять историю данных. Исключения (например, справочники без softDeletes или служебные таблицы) явно указаны в описании соответствующих сущностей.

### **Базовые справочники (Dictionaries)**

Искусственному интеллекту в рамках AI-пайплайна категорически запрещено генерировать или галлюцинировать новые категории; алгоритм обязан осуществлять маппинг (сопоставление) извлеченных семантических смыслов исключительно на утвержденные идентификаторы справочников. Это гарантирует, что пользовательские фильтры на фронтенде будут работать предсказуемо.

### **Полиморфная архитектура: Организаторы и Инициативы**

Для реализации абстракции организатора вводится таблица-маршрутизатор organizers. Этот паттерн избавляет систему от необходимости создавать отдельные таблицы связей (content\_user, content\_group) и позволяет привязывать отзывы, рейтинги и мероприятия к единому идентификатору.

| Таблица (Сущность) | Структура полей | Назначение и контекст |
| :---- | :---- | :---- |
| organizers | id (UUID, PK), organizable\_type (string, index), organizable\_id (UUID, index), contact\_phones (JSONB), contact\_emails (JSONB), status (string, index). | Универсальная точка входа. Поле organizable\_type хранит строковой алиас (Morph Map) целевой модели. Использование типа JSONB для контактов позволяет хранить массивы номеров, очищенных алгоритмами Dadata, без избыточных JOIN-запросов. |
| organizations | id (UUID, PK), title (string), description (text), inn (string, unique, index), ogrn (string, unique, index), site\_urls (JSONB), organization\_type\_id (FK), ownership\_type\_id (FK), coverage\_level\_id (FK), works\_with\_elderly (boolean, index), ai\_confidence\_score (decimal, 8,4), ai\_explanation (text), ai\_source\_trace (JSONB), target\_audience (JSONB), vk\_group\_id (bigInteger, nullable, index), ok\_group\_id (bigInteger, nullable, index), status (string, index), timestamps, softDeletes. | Хранилище профилей формальных юридических лиц. Поля inn и ogrn проходят предварительную очистку от нецифровых символов на этапе парсинга перед отправкой в API Dadata. Поля vk\_group\_id и ok\_group\_id используются для интеграции с мини-приложениями ВКонтакте и Одноклассников. |
| initiative\_groups | id (UUID, PK), name (string), description (text), community\_focus (string), established\_date (date, nullable). | Хранилище профилей неформальных коллективов (например, «группа бабушек в местном парке»), не имеющих юридических реквизитов. |

Важнейшим аспектом сущности organizations является интеграция специфических полей для взаимодействия с AI-пайплайном. Процесс модерации и обогащения данных требует сохранения не только финального вердикта искусственного интеллекта, но и метрик его уверенности, а также цепочки рассуждений.

К сущностям organizations и initiative\_groups добавляются следующие критические поля:

* works\_with\_elderly (boolean): Бинарный флаг, подтверждающий, что не менее 30% активности субъекта направлено на помощь старшему поколению. Это фундаментальный критерий отсева нерелевантных инициатив, сфокусированных исключительно на детях или экологии.  
* ai\_confidence\_score (decimal, 8,4): Численное выражение вероятности (от 0.0 до 1.0) того, что сгенерированная классификация и принятое решение («accepted»/«rejected») являются корректными. Подобные метрики повсеместно применяются в системах предиктивной аналитики и модерации контента для балансировки между автоматическим пропуском и ручной проверкой. Концепции с показателем уверенности ниже заданного порога маршрутизируются на ручную верификацию.  
* ai\_explanation (text): Текстовое обоснование решения. Структурированные выходные данные LLM позволяют системе сохранить логику вывода в формате, пригодном для аудита модератором.  
* ai\_source\_trace (JSONB): Детализированный массив объектов, связывающий сущность с таблицей источников. Фиксирует откуда и когда были получены подтверждающие данные (подробное описание структуры см. в разделе Source Manager).

#### **Расширение модели: Физические лица (Individuals)**

Сущность Individual была заявлена в полиморфной карте, но требует строгой спецификации. В рамках модели individuals выступает одной из разрешаемых сущностей для organizers. Для минимизации рисков, связанных с обработкой персональных данных (PII) и соблюдением законодательства (ФЗ-152), набор полей максимально ограничен: id (UUID, PK), full\_name (string), role (string), contact\_email (string, nullable), contact\_phone (string, nullable), consent\_given (boolean, обязательное подтверждение согласия). Эти лица могут выступать организаторами разовых активностей или контактными лицами неформальных инициативных групп.1

#### **Дополнительные атрибуты: Целевая аудитория (Target Audience)**

Несмотря на наличие флага работы с пожилыми людьми (works\_with\_elderly), для расширенной фильтрации и аналитики в сущности organizations и events добавляется поле target\_audience (JSONB, nullable). Оно реализуется как массив объектов или ссылок на справочник target\_audience. Возможные значения охватывают: пожилые люди, родственники, профильные специалисты, молодежь/дети (для межпоколенческих инициатив). Это позволяет точнее маршрутизировать контент и услуги. Использование JSONB обеспечивает гибкость структуры данных и возможность расширения без изменения схемы БД.

### **Управление локациями: Сущность Venue и Геометрия PostGIS**

Для решения проблем с геоданными вводится отдельная сущность venues (Площадки). Связь реализуется через pivot-таблицы (M:N) organization\_venues и event\_venues.

| Сущность | Поля и типы данных | Архитектурное обоснование |
| :---- | :---- | :---- |
| venues | id (UUID, PK), address\_raw (string), fias\_id (string, index), kladr\_id (string, index), region\_iso (string, index), coordinates (Geometry Point, SRID 4326), timestamps, softDeletes. | Поля fias\_id и kladr\_id обеспечивают синхронизацию с Федеральной информационной адресной системой через Dadata, что гарантирует абсолютную точность идентификации объектов. Поле coordinates использует специализированный тип данных PostGIS для хранения точки в проекции EPSG:4326. Индекс на region\_iso используется для региональной фильтрации. |

#### **Прямая связь мероприятий и организаций (Денормализация)**

В таблицу events (или на уровне ER-описания) внедряется дополнительное поле organization\_id (nullable, FK на organizations). Хотя у мероприятия уже есть organizer\_id (указывающий на полиморфную таблицу), прямое поле organization\_id вводится для упрощения и ускорения наиболее частых SQL-запросов (например, «вывести все мероприятия конкретного юридического лица»). Это избавляет от необходимости делать ресурсоемкие JOIN-запросы через таблицу-маршрутизатор organizers в тех случаях, когда организатором выступает формальная организация.1

### **Связующие матрицы (Pivot Tables)**

* organization\_problem\_categories: (organization\_id, problem\_category\_id). Отражает исторический массив hp\_listing\_category.  
* organization\_services: (organization\_id, service\_id). Отражает массив hp\_listing\_service.  
* **event\_event\_categories**: (event\_id, event\_category\_id). Pivot таблица для связи событий с категориями. Название изменено на event\_event\_categories для избежания конфликта с справочником event\_categories.  
* event\_venues: (event\_id, venue\_id). Событие может транслироваться из нескольких точек (смешанный формат).  
* organization\_venues: (organization\_id, venue\_id, is\_headquarters boolean). Фиксирует филиальную сеть.

### **Контентная модель (Статьи и Руководства)**

Модель данных расширяется для поддержки редакционного контента платформы. Вводится сущность articles (или проектируется слой маппинга для существующих постов WordPress). Данная структура жизненно необходима для реализации концепции «Answer Instead of Search»: она позволяет искусственному интеллекту или рекомендательной системе платформы автоматически связывать образовательные руководства (например, о деменции) с карточками соответствующих центров и услуг.

**Статусы и управление жизненным циклом (State Machine)**

В условиях автоматизированного наполнения каталога качество данных становится главной уязвимостью. Жизненный цикл сущности контролируется полем status.

| Статус (Состояние) | Описание и триггеры | Действия системы и видимость |
| :---- | :---- | :---- |
| draft (Черновик) | Исходное состояние при поступлении сырых данных до обработки AI-пайплайном. | Сущность скрыта от публичных API. Ожидает запуска обогащения. |
| pending\_review (На модерации) | Назначается, если AI вынес решение accepted, но уровень уверенности ai\_confidence\_score оказался ниже установленного порога. | Сущность скрыта от публики. Попадает в очередь панели администратора для гибридной ручной модерации. |
| approved (Опубликовано) | Присваивается автоматически (Smart Publish), если AI высоко уверен в релевантности (confidence \>= 0.85, флаг works\_with\_elderly \= true). | Сущность доступна во всех публичных REST API. |
| rejected (Отклонено) | Алгоритм или модератор определил несоответствие критериям. | Сущность перемещается в "холодное" хранилище. |
| archived (Архивировано) | Присваивается историческим записям (организация закрылась). | Исключается из основной выдачи. |

### **3\. Интеграция с Laravel Scheduler (Оркестрация)**

В версии 1.1 планировщик задач переносится из Python/Celery полностью в экосистему Laravel (Navigator Core). Это делает Laravel единым источником истины для расписаний.

**Алгоритм оркестрации:**

1. **Core Scheduler:** Крон-задача Laravel (schedule:run) ежечасно проверяет таблицу sources.  
2. **Pull Query:** Запрос GET /api/internal/sources?due=true (или прямой вызов Eloquent) выбирает источники, где last\_crawled\_at \+ crawl\_period\_days \<= NOW().  
3. **Push to Harvester:** Core формирует пакет задач и отправляет POST-запрос микросервису Harvester (на Python/Go):  
   POST /harvest/run  
   Body: {"sourceIds": \["uuid1", "uuid2"\]}  
4. Harvester выполняет обход, а результаты отправляет в AI-Enrichment.

*Преимущества (Pros):* Полный контроль над очередями в одном приложении (Laravel Horizon), легкая интеграция с бизнес-логикой и UI-админкой.

*Недостатки (Cons):* Необходимость создания надежного механизма webhook-ответов от Harvester, чтобы Core знал об успешном завершении задачи или тайм-аутах.

### **4\. API для модераторов (Source Manager UI)**

Для управления подсистемой через Laravel Nova / Filament проектируются следующие внутренние эндпоинты:

* GET /api/internal/sources — Получение списка источников с фильтрами по статусу (last\_status), типу (kind) и региону.  
* PATCH /api/internal/sources/{id} — Ручное изменение параметров (увеличение crawl\_period\_days, приостановка priority).  
* GET /api/internal/source-proposals — Вывод предложений LLM-агента о добавлении новых источников (сформированных из парсинга ссылок внутри текстов). Выводит confidence\_score, reason и предложенный JSON parse\_profile.  
* POST /api/internal/source-proposals/batch-accept — Массовое утверждение (batch) источников, например, после импорта реестра президентских грантов.

**Архитектура REST API (Спецификация контрактов)**

Архитектура прикладного программного интерфейса Navigator Core строится по принципу API-first и разделяется на два изолированных контура: публичный (Public B2C) и внутренний (Internal System/AI).

Публичный контур разрабатывается с учетом жестких требований к интеграции в мини-приложения ВКонтакте (VK Mini Apps) и Одноклассников (OK Mini Apps). Ограничения мобильных сетей диктуют необходимость минимизации размера ответа (Payload Size) и исключения избыточных данных (отсутствие HTML-тегов в текстовых полях, строгая кодировка UTF-8). Для обеспечения производительности все GET-запросы публичного API кэшируются на уровне Redis и используют предварительно загруженные отношения (Eager Loading) в Laravel для избежания проблемы N+1 запросов.

### **Публичный контур (Веб-платформа и VK/OK Mini Apps)**

**Специфика интеграции с VK/OK Mini Apps и Идентификаторы:** В целях глубокой интеграции с социальными сетями в сущности organizations и sources добавляются поля для хранения социальных идентификаторов: vk\_group\_id и ok\_group\_id.1 Публичный API жестко разделяется на два типа эндпоинтов:

* **Легковесные списочные (List API):** Используют компактные DTO (Data Transfer Objects), возвращающие исключительно необходимые для отрисовки карточек списков поля (id, title, cover\_image, coordinates). Это критически важно для минимизации Payload Size и экономии трафика в мобильных сетях при работе с Mini Apps.1  
* **Детальные (Detail API):** Возвращают полную информацию только при переходе внутрь конкретной сущности.1

**Стратегия миграции и различия с текущей JSON‑схемой**

Текущая реализация системы (отраженная в import.txt и navigator\_listing\_schema\_copy.json) является плоской (flat schema), что свойственно парадигме метаданных (post\_meta) в WordPress. Этот подход позволяет быстро запускать MVP, но приводит к катастрофической деградации производительности при масштабировании базы данных, так как любые фильтры требуют ресурсоемких SQL-запросов с множественными JOIN-соединениями к одной и той же таблице метаданных. Переход на нормализованную реляционную структуру Navigator Core требует глубокой реструктуризации существующих данных.

### **Трансформация существующих полей**

1. **Прямая миграция (с изменением типов)**:  
   * Поля post\_title и post\_content конвертируются в title и description соответственно, очищаясь от рудиментарных HTML-сущностей, не поддерживаемых в мобильных приложениях.  
   * Реквизиты hp\_inn и hp\_ogrn мигрируют в строго типизированные уникальные строковые столбцы таблицы organizations с предварительной очисткой от пробелов и дефисов.  
2. **Глубокая декомпозиция (Дробление на сущности)**:  
   * **Геолокация и адреса:** Ранее координаты хранились как разрозненные строки (hp\_latitude, hp\_longitude), что приводило к программным ошибкам (в скрипте import.txt прямо указано: "Исправлена обработка координат: широта и долгота меняются местами"). Поля hp\_location, hp\_region и hp\_town также дублировали информацию. В новой модели вся эта группа полей полностью удаляется из родительских таблиц. Данные мигрируют в нормализованную таблицу venues, где координаты инкапсулируются в защищенный бинарный тип Geometry(Point) СУБД PostGIS. Это делает невозможным некорректное чтение осей и открывает доступ к мощным гео-функциям.  
   * **Контактная информация:** Поля hp\_phone, hp\_email и hp\_site в WordPress объединялись в единую строку через запятую (implode(', ', $value)). Эта практика делает невозможным поиск по конкретному номеру или валидацию формата. В новой схеме они трансформируются в столбцы типа JSONB (contact\_phones, contact\_emails), что сохраняет гибкость хранения множественных контактов, но возвращает возможность индексации и структурированного доступа.  
3. **Трансформация в реляционные связи (Many-to-Many)**:  
   * Данные, хранившиеся как строки вида "Проблемы с памятью и когнитивные нарушения \- 7" в массивах hp\_listing\_category, hp\_listing\_service, hp\_listing\_type, больше не хранятся в виде текстовых строф. При миграции скрипт извлекает числовой идентификатор (например, 7\) и создает физическую связь (FK) в сводной таблице organization\_problem\_categories, связывающей организацию со справочником problem\_categories. Это обеспечивает ссылочную целостность и исключает риск опечаток.

### **Фундаментальные архитектурные добавления**

Новая доменная модель не просто реструктурирует старые данные, но и вводит концепции, принципиально отсутствовавшие в экосистеме HivePress:

* **Инкапсуляция времени (Events & RRule)**: Ранее система не могла адекватно описывать активности с датами и расписаниями, поскольку профиль организации статичен. Внедрение графа Event \-\> EventInstance с поддержкой стандартов RFC 5545 является мощнейшим апгрейдом платформы.  
* **Полиморфные организаторы (Polymorphism)**: HivePress вынуждал создавать "Организацию" даже для неформальной группы из трех человек. Полиморфная модель organizers решает эту проблему элегантно, позволяя системе работать с сущностями разной правовой природы.  
* **Материализация метрик AI**: Показатели работы алгоритмов (works\_with\_elderly, ai\_confidence\_score), которые ранее существовали только транзитно в оперативной памяти Python-скрипта, теперь являются первоклассными гражданами базы данных. Это позволяет строить сложную бизнес-логику поверх метрик машинного обучения.

**Рекомендации для реализации в инфраструктуре Laravel**

Успешная материализация спроектированной модели требует от разработчиков строгого соблюдения лучших практик экосистемы Laravel и использования мощных возможностей СУБД. Наивный перенос паттернов из WordPress приведет к быстрому коллапсу производительности.

### **Аналитика и агрегация метрик**

Для поддержки дашбордов и статистических отчетов вводится разделение на транзакционную (OLTP) и аналитическую модели данных. Чтобы не перегружать боевые таблицы (events, organizations) постоянными агрегатными вычислениями (например, подсчет общего охвата, количества мероприятий по регионам), проектируются выделенные таблицы для статистических агрегатов (например, organization\_stats, regional\_activity\_stats). Данные в них обновляются асинхронно через фоновые задачи (Jobs) или материализованные представления PostgreSQL, аккумулируя такие показатели, как количество проведенных мероприятий, размер аудитории и динамика региональной активности.1

### **Инфраструктура баз данных: PostgreSQL и PostGIS**

Настоятельно рекомендуется использовать **PostgreSQL в связке с расширением PostGIS**, полностью исключив MySQL или MariaDB из технологического стека.

* **Пространственная геометрия:** При масштабировании базы данных до миллионов геокоординат и необходимости фильтрации в реальном времени (например, при перемещении карты в мобильном приложении) R-tree индексы MySQL демонстрируют недостаточную стабильность и узость функционала. PostGIS предоставляет индексы типа GiST и функционал БР-деревьев (BRIN), способные обрабатывать огромные геопространственные массивы данных с миллисекундной задержкой.  
* **Миграции Laravel:** Для корректной инициализации таблиц с пространственными данными необходимо использовать специфичные методы Blueprint:  
  PHP  
  // В миграции для таблицы venues  
  Schema::create('venues', function (Blueprint $table) {  
      $table\-\>uuid('id')-\>primary();  
      $table\-\>string('address\_raw');  
      $table\-\>string('fias\_id')-\>index();  
      // Создание поля геометрии с использованием SRID 4326 (WGS 84\)  
      $table\-\>geometry('coordinates', 'point', 4326);  
      $table\-\>timestamps();  
  });  
  // Добавление пространственного индекса GiST  
  DB::statement('CREATE INDEX venues\_coordinates\_index ON venues USING GIST (coordinates);');

* **Оптимизация запросов (Eloquent):** Запрещается вычисление расстояний на стороне приложения (в коллекциях PHP) или использование сложных тригонометрических математических выражений в чистом SQL, вызывающих полное сканирование (full table scan). Следует использовать нативные функции PostGIS через сырые выражения (Raw Expressions) или специализированные пакеты (например, grimzy/laravel-mysql-spatial или аналоги для Postgres):  
  PHP  
  Venue::whereRaw("ST\_DWithin(coordinates, ST\_MakePoint(?,?)::geography,?)", \[$lng, $lat, $radiusInMeters\])-\>get();

### **Стратегия работы с повторяющимися событиями (RRule)**

Вычислительная сложность парсинга строк RFC 5545 (RRule) на лету делает невозможным их использование в SQL-условиях (WHERE).

* **Материализация через Task Scheduling:** Приложение должно использовать мощную систему планирования задач Laravel (Task Scheduling). При создании или обновлении события Event, модель инициирует асинхронную задачу (Job), которая использует PHP-парсер (например, rlanvin/php-rrule или simshaun/recurr) для вычисления всех дат вхождений (instances) на следующие 90 дней и пакетно вставляет их в таблицу event\_instances.  
* Ежедневная крон-команда (php artisan schedule:run) должна проверять все активные RRule и догенерировать экземпляры, поддерживая скользящее окно (rolling window) в 90 дней в будущем.  
* Это позволяет свести сложнейшую логику календаря к примитивному, но невероятно быстрому Eloquent-запросу с использованием B-Tree индексов:  
  PHP  
  EventInstance::with(\['event.organizer', 'venue'\])  
               \-\>where('status', 'scheduled')  
               \-\>whereBetween('start\_datetime',)  
               \-\>get();

  Использование методов with() критично для загрузки связей (Eager Loading), что предотвращает проблему множественных запросов к БД при сериализации данных в JSON.

### **Полиморфные отношения и безопасность**

* **Morph Map:** По умолчанию Laravel использует полные пространства имен PHP-классов (например, App\\Models\\Organization) в колонке organizable\_type. Это создает жесткую связность базы данных и кода, что усложняет рефакторинг. Рекомендуется зарегистрировать алиасы в AppServiceProvider:  
  PHP  
  use Illuminate\\Database\\Eloquent\\Relations\\Relation;

  Relation::enforceMorphMap(\[  
      'Organization' \=\> 'App\\Models\\Organization',  
      'InitiativeGroup' \=\> 'App\\Models\\InitiativeGroup',  
      'Individual' \=\> 'App\\Models\\Individual',  
  \]);

  Метод enforceMorphMap гарантирует, что разработчики не смогут случайно использовать полные классы в базе данных.

### **Использование AI-ассистентов (Cursor / Copilot) для кодогенерации**

При передаче спроектированной доменной модели инструментам кодогенерации на базе LLM (например, IDE Cursor), рекомендуется применять стратегию пошаговой детализации (Incremental Prompting):

1. **Справочники (Dictionaries):** Сначала запросите генерацию миграций и фабрик для всех базовых справочников. Укажите ИИ, что таблицы справочников не должны содержать полей временных меток ($table-\>timestamps()), чтобы сэкономить ресурсы.  
2. **Полиморфизм:** Инструктируйте ИИ создать миграцию organizers с колонками $table-\>uuidMorphs('organizable'). Запросите генерацию соответствующих трейтов или методов morphTo() и morphMany() в Eloquent-моделях.  
3. **Геометрия:** Прямо укажите ассистенту: "Использовать методы PostGIS геометрии для колонки координат в таблице venues. Не использовать обычные decimal(10,8) колонки для широты и долготы".  
4. **Слои API:** Потребуйте разделить логику сериализации. Инструктируйте Cursor создать классы API Resources (например, OrganizationResource), которые будут жестко форматировать и обрезать данные для публичного контура, исключая случайную утечку чувствительных полей (например, ai\_confidence\_score или contact\_emails) в клиентские приложения.

Реализация спроектированной доменной модели и следование архитектурным предписаниям позволит Navigator Core преодолеть ограничения старой инфраструктуры. Выделение логики AI-проверки, строгая нормализация адресов через Dadata и глубокая интеграция с геопространственными инструментами создадут надежный, отказоустойчивый и масштабируемый фундамент для развития «Навигатора здорового долголетия» в федеральную цифровую экосистему.
