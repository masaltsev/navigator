# **Проектирование архитектуры и маппинга данных: Переход от WordPress (HivePress) к Navigator Core**

## **Инфраструктурный контекст и стратегическое видение проекта**

Проект «Навигатор здорового долголетия» находится на этапе глубокой структурной трансформации, обусловленной переходом от устаревшей монолитной архитектуры на базе системы управления контентом WordPress и плагина каталогизации HivePress к высокопроизводительному, масштабируемому фреймворку Laravel, который получил внутреннее название Navigator Core. Данная техническая миграция не является исключительно программным рефакторингом; она представляет собой фундаментальный сдвиг в самой парадигме предоставления информационных услуг. Исторически сложившаяся инфраструктура была спроектирована вокруг концепции простого поиска по каталогу, где пользователь самостоятельно фильтровал реестр организаций. Целевая архитектура Navigator Core реализует инновационную концепцию «Answer Instead of Search» (Ответ вместо поиска), которая предполагает алгоритмическое связывание конкретной проблемы пользователя с релевантными образовательными статьями и верифицированными поставщиками услуг.1

Существующая база данных WordPress аккумулировала значительный объем информации, однако анализ выявляет критические ограничения этого набора данных. В текущей базе данных представлены преимущественно формальные организации, то есть зарегистрированные юридические лица, в то время как неформальные инициативные группы и локальные сообщества практически отсутствуют.1 Кроме того, текстовые описания этих организаций в устаревшей системе отличаются краткостью, фрагментарностью и во многих случаях содержат неактуальную информацию.1 Подобное качество текстовых данных делает их абсолютно непригодными в качестве надежного первоисточника для автоматизированного построения событийной архитектуры (Events) и генерации динамического контента, требуемого новой системой.1

В связи с этими ограничениями, процесс миграции проектируется как сложный, многоступенчатый пайплайн, разделенный на фазы первичного извлечения и последующего глубокого обогащения. На первом этапе требуется осуществить строгий перенос данных из нереляционной структуры метаданных WordPress в жестко типизированную схему Navigator Core, включая подготовку таблиц для отслеживания источников информации. На втором этапе активируется интеллектуальная система обогащения, которая использует поисковые алгоритмы, интеграции с внешними API и большие языковые модели для расширения профиля каждого объекта.1 Настоящий документ представляет собой исчерпывающее руководство по маппингу данных, анализу исходных структур, проектированию целевой модели и стратегическому планированию этапов миграции.

## **Архитектурный анализ исходных данных WordPress и HivePress**

Система WordPress исторически опирается на паттерн проектирования Entity-Attribute-Value (EAV), который обеспечивает максимальную гибкость при добавлении новых полей без изменения схемы базы данных, однако создает серьезные проблемы с производительностью при выполнении сложных аналитических запросов и агрегаций.1 В контексте проекта «Навигатор» все организации представлены в виде пользовательского типа записей (Custom Post Type), который носит идентификатор hp\_listing.1 Базовые атрибуты организации, такие как ее наименование и основное описание, хранятся в стандартных столбцах post\_title и post\_content таблицы wp\_posts.1 Тем не менее, вся специфика бизнес-логики, включая контактные данные, юридические реквизиты и геопозиционирование, вынесена в таблицу wp\_postmeta, что требует тщательного анализа перед началом миграции.

### **Структура метаданных и идентификация сущностей**

Метаданные каждой записи hp\_listing формируют профиль организации. Ключевыми идентификаторами в этой системе выступают мета-ключи hp\_inn (Идентификационный номер налогоплательщика) и hp\_ogrn (Основной государственный регистрационный номер). Эти целочисленные значения исторически использовались скриптами импорта в качестве первичных ключей для дедубликации записей; алгоритм поиска существующих листингов формировал запрос WP\_Query с логическим оператором OR по этим двум полям, чтобы определить, требуется ли создание новой записи или обновление существующей.1

Помимо идентификаторов, схема данных `docs/archive/wp-schema/navigator_listing_schema_copy.json` регламентирует наличие контактной информации. Поля hp\_phone, hp\_email и hp\_site изначально декларируются как массивы строк, однако в базе данных WordPress они зачастую хранятся в виде сериализованных строк или текста, разделенного запятыми.1 Скрипт импорта, обрабатывающий эти массивы, содержал логику фильтрации, которая удаляла значения, равные строке «None» или нулевым значениям, после чего объединяла оставшиеся элементы в единую строку через запятую перед сохранением в базу данных.1 Это означает, что миграционный скрипт Navigator Core должен будет выполнить обратную операцию: разбить эти строки, очистить их от пробелов и сформировать валидные массивы JSON для новой реляционной модели.

Особого внимания при анализе исходных данных требует механизм работы с географическими координатами. За физический адрес отвечает строковое поле hp\_location, которое является обязательным, а также вспомогательное поле hp\_town для указания населенного пункта и hp\_region для хранения числового идентификатора региона.1 Однако поля hp\_longitude и hp\_latitude подверглись программной инверсии в историческом коде. Анализ исходного кода скрипта import.php выявляет наличие преднамеренной замены: значение, поступающее как долгота, сохранялось в мета-ключ hp\_latitude, а значение широты записывалось в мета-ключ hp\_longitude.1 Данная аномалия, вероятно, была внедрена для компенсации ошибки в стороннем картографическом сервисе или API, однако при миграции в математически строгую систему Navigator Core эту инверсию необходимо нивелировать, вернув значениям их истинный географический смысл. Дополнительно в базе присутствуют мета-ключи hp\_ceo (имя руководителя организации) и hp\_dobro\_ru (целочисленный идентификатор профиля на федеральном портале волонтеров), которые также подлежат переносу.1

### **Семантическая структура таксономий и классификаторов**

Каталогизация в HivePress опирается на стандартный механизм таксономий WordPress. Классификаторы в проекте «Навигатор» обладают высокой степенью детализации и используются для связи организаций с конкретными медицинскими проблемами, социальными услугами и правовыми формами. Архитектура старого импорта реализовывала весьма специфический подход к привязке терминов: вместо поиска терминов по их буквенному названию, система ожидала получить строку, в которой числовой идентификатор термина был добавлен в самый конец после дефиса.1 Регулярное выражение preg\_match('/- (\\d+)$/') извлекало эти цифры, которые затем использовались в функции wp\_set\_object\_terms для формирования связей в базе данных.1

Анализ словарей таксономий выявляет четыре ключевых измерения данных, которые необходимо перенести в новые реляционные справочники:

Первое измерение — это таксономия категорий (hp\_listing\_category), которая фактически описывает проблематику целевой аудитории. Эта таксономия включает в себя медицинские диагнозы, физические состояния и социальные вызовы. Например, проблемы с памятью и когнитивные нарушения идентифицируются кодом 7, остеопороз и переломы — кодом 8, хроническая боль — кодом 10, а потеря автономности и одинокое проживание — кодами 18 и 20 соответственно.1 Эти данные критически важны для новой системы маршрутизации контента.

Второе измерение — таксономия услуг (hp\_listing\_service), представляющая собой исчерпывающий перечень решений, предоставляемых организациями. Классификатор варьируется от стационарного ухода (например, психоневрологические интернаты под кодом 65 или стационары для людей с деменцией под кодом 112\) до конкретных медицинских специалистов, таких как гериатры (код 143), неврологи (код 142\) и сурдологи (код 122).1 Кроме того, таксономия включает социальные и бытовые услуги: сиделки (код 110), доставка продуктов (код 109\) и адаптация жилых помещений (код 105).1

Третье измерение — типы организаций (hp\_listing\_type), характеризующие функциональный профиль юридического лица. Здесь выделяются такие категории, как больницы и госпитали (код 33), гериатрические центры (код 30), комплексные центры социального обслуживания (код 40), благотворительные организации (код 43\) и центры серебряного волонтерства (код 174).1

Четвертое измерение — форма собственности (hp\_listing\_ownership), определяющая юридический статус организации. Данный словарь жестко регламентирует правовые формы в соответствии с законодательством Российской Федерации. К государственному и муниципальному сектору относятся государственные бюджетные учреждения (ГБУ, код 154), государственные казенные учреждения (ГКУ, код 159\) и муниципальные бюджетные учреждения (МБУ, код 153).1 Некоммерческий сектор представлен фондами (код 161\) и автономными некоммерческими организациями (АНО, код 162), а коммерческий — обществами с ограниченной ответственностью (ООО, код 163).1

Все эти таксономии в WordPress существуют в виде плоских списков терминов. При переходе на архитектуру Navigator Core эти данные должны быть преобразованы в строгие реляционные связи, что требует детального проектирования правил отображения данных (маппинга).

## **Проектирование модели маппинга: WordPress → Navigator Core**

Переход на фреймворк Laravel сопровождается глубокой нормализацией базы данных. Ключевой архитектурный шаг заключается в декомпозиции монолитной сущности hp\_listing. В устаревшей парадигме информация о юридическом лице неразрывно хранилась вместе с информацией о его географическом положении. В парадигме Navigator Core вводится четкое разделение: таблица organizations отвечает за глобальные атрибуты юридического лица (название, ИНН, форма собственности, общие контакты), а таблица venues хранит информацию о физических филиалах, точках присутствия и их координатах. Это позволяет одной организации иметь множество адресов оказания услуг.1

Дополнительно целевая модель внедряет новые системные таблицы для поддержки интеллектуального конвейера обработки данных (AI Pipeline). Таблицы sources и parse\_profiles предназначены для регистрации происхождения каждой единицы информации, а таблица articles выступает фундаментом для реализации образовательного вектора системы.1

### **Маппинг сущности Organizations**

Таблица organizations становится центральным узлом для всех юридических лиц. Процесс переноса данных в эту таблицу требует применения специфических алгоритмов очистки и валидации.

| :---- | :---- | :---- | :---- |
| wp\_posts.post\_title | name | string | Прямое копирование значения с применением функции trim() для устранения начальных и конечных пробелов. Представляет официальное наименование организации. |
| wp\_posts.post\_content | description | text | Перенос исходного HTML-контента. Данное поле помечается как требующее ревизии, так как исходные тексты признаны слишком короткими и неактуальными. На Этапе 1 контент будет переписан большой языковой моделью (LLM).1 |
| wp\_postmeta.hp\_inn | inn | string | Критическое поле для идентификации. Значение извлекается, приводится к строковому типу, после чего удаляются все нецифровые символы. На уровне базы данных устанавливается индекс для ускорения поиска. ### **Маппинг сущности Venues (Филиалы и локации)**

В устаревшей системе WordPress концепция филиалов отсутствовала; каждая запись hp\_listing могла иметь только один адрес. При миграции для каждой созданной организации будет алгоритмически сгенерирована как минимум одна запись в таблице venues, связанная с ней через внешний ключ.

| :---- | :---- | :---- | :---- |
| Автоматическая генерация | organization\_id | unsignedBigInteger | Внешний ключ (Foreign Key), устанавливающий жесткую реляционную связь с родительской организацией, созданной на предыдущем шаге. |
| Автоматическая генерация / post\_title | name | string | Наименование филиала. В рамках сырой миграции целесообразно использовать название родительской организации с добавлением суффикса, например, «Главное подразделение». |
| wp\_postmeta.hp\_location | address | string | Прямой перенос текстового представления адреса, введенного пользователем или геокодером.1 |
| wp\_postmeta.hp\_town | city | string, nullable | Строка населенного пункта переносится с применением регулярных выражений для очистки от административных префиксов. ### **Маппинг справочников и связующих таблиц (Pivot Tables)**

В архитектуре Laravel связи типа «многие ко многим» реализуются через промежуточные таблицы (pivot tables). Это обеспечивает высокую скорость выборок и ссылочную целостность базы данных. Старые таксономии WordPress конвертируются в следующие структуры:

| Исходная таксономия WP | Целевая таблица Core (Справочник) | Промежуточная таблица связей (Pivot) | Алгоритм миграции |
| :---- | :---- | :---- | :---- |
| hp\_listing\_category | problem\_categories | organization\_problem\_category | Идентификаторы медицинских и социальных проблем (от 7 до 29\) извлекаются из связей WordPress. В новой системе создается связь между ID организации и ID проблемы.1 |
| hp\_listing\_service | services | organization\_service | Массив идентификаторов услуг (от 65 до 143\) используется для заполнения связующей таблицы, регламентирующей, какие конкретно сервисы предоставляет организация.1 |
| hp\_listing\_type | organization\_types | organization\_type\_pivot | Идентификаторы профиля деятельности (от 30 до 175\) определяют типологию организации в новой реляционной модели.1 |

### **Маппинг сущностей контента и экосистемы обогащения данных**

Переход на систему Navigator Core сопровождается внедрением совершенно новых таблиц, которые формируют инфраструктуру для интеллектуальной обработки контента и машинного обучения.

**Сущность articles (Статьи и образовательные материалы)** В рамках стратегического сдвига к предоставлению готовых ответов («Answer Instead of Search»), контентные статьи становятся полноправными участниками доменной модели. Если в исторической базе WordPress присутствовали записи типа post (стандартный формат блогов), они должны быть перенесены в таблицу articles. Согласно спецификациям Laravel, эта таблица обязана включать стандартные временные метки created\_at и updated\_at для отслеживания жизненного цикла контента, а также поддержку мягкого удаления через поле deleted\_at (softDeletes), что позволяет сохранять историческую целостность системы при удалении материалов.1 Статьи будут алгоритмически связываться с организациями, позволяя системе рекомендовать пользователю конкретных врачей или центры ухода непосредственно во время чтения материалов по релевантной тематике.1

**Сущности sources (Источники) и parse\_profiles (Профили парсинга)** Интеллектуальный конвейер (Smart Pipeline) нуждается в строгом учете происхождения информации. Таблица sources представляет собой реестр авторизованных источников данных. В процессе первоначальной миграции скрипт должен создать в этой таблице базовую системную запись, представляющую старую базу данных WordPress. В дальнейшем эта таблица будет пополняться ссылками на государственные реестры, официальные сайты организаций и социальные сети.1 В свою очередь, таблица parse\_profiles не имеет аналогов в старой системе; она предназначена для хранения конфигурационных файлов и пресетов краулеров. Эти профили определяют, по каким CSS-селекторам или API-эндпоинтам система должна собирать новую информацию с внешних сайтов для постоянной актуализации карточек организаций. Заполнение этой таблицы происходит не в момент переноса данных, а на этапе настройки интеллектуального обогащения.

## **Двухэтапная стратегия трансформации и миграции данных**

Учитывая неудовлетворительное качество текущих текстовых описаний и отсутствие структурированных данных для реализации событийной архитектуры, прямая выгрузка базы данных в продакшен невозможна.1 Процесс разбит на два последовательных и строго регламентированных этапа, что минимизирует риски деградации сервиса и обеспечивает плавное наращивание качества данных перед масштабной рекламной кампанией 2026 года.1

### **Этап 0: Базовая инфраструктурная миграция (Raw Migration)**

Цель нулевого этапа заключается в механическом и синтаксическом переносе исторической информации из структуры EAV (WordPress) в реляционную систему (PostgreSQL/MySQL под управлением Laravel). На этом этапе не производится генерация новых текстов или поиск данных в интернете; главной задачей является обеспечение целостности и структурной правильности имеющейся информации.

**Последовательность действий на Этапе 0:**

1. **Инициализация справочников:** Запуск механизмов наполнения базы данных (Database Seeders) в Laravel. В базу заносятся все статические эталонные значения из словарей категорий, сервисов, типов организаций и форм собственности.1 Для сохранения совместимости и упрощения логики переноса рекомендуется сохранить оригинальные идентификаторы терминов (например, код 143 должен остаться закрепленным за услугой «Гериатр»).  
2. **Агрегация и извлечение данных:** Выполнение SQL-запросов к устаревшей базе данных. Поскольку метаданные хранятся в отдельных строках таблицы wp\_postmeta, извлечение должно использовать функции агрегации (на стороне СУБД) для формирования плоских объектов, готовых к процессу маппинга.  
3. **Применение трансформаций (ETL):** Данные прогоняются через логические фильтры. Выполняется десериализация контактных данных (телефонов, email-адресов) и их упаковка в JSON. Производится критически важная замена координат местами для устранения исторического бага.1 Очищаются идентификационные поля (ИНН, ОГРН). Важно отметить, что если у организации отсутствует ИНН или ОГРН, скрипт не должен отклонять такую запись; она переносится в новую базу, но маркируется специальным флагом, указывающим на острую необходимость обогащения.  
4. **Создание системных записей:** В таблицу sources добавляется запись, легитимизирующая происхождение данных.  
5. **Пакетная загрузка (Bulk Load):** Массив обработанных данных загружается в таблицы organizations и venues пакетами, чтобы не вызывать перегрузку оперативной памяти. Параллельно заполняются промежуточные таблицы для связи организаций с их услугами и категориями.  
6. **Маркировка происхождения:** Каждой созданной организации в поле ai\_source\_trace записывается JSON-объект, содержащий информацию о дате миграции и ссылающийся на системный источник «WordPress Legacy».1

По завершении Этапа 0 система получает высокопроизводительную реляционную базу данных. Однако с содержательной точки зрения организации остаются неполными, а их текстовые описания по-прежнему непригодны для реализации глубокой бизнес-логики и построения событий.1 Этот недостаток устраняется на следующем шаге.

### **Этап 1: Интеллектуальное обогащение и расширение экосистемы источников**

Первый этап знаменует собой активацию Smart Pipeline — системы автоматизированного краулинга, интеграции API и машинного обучения. Именно этот этап трансформирует сухой реестр юридических лиц в живую, динамически обновляемую экосистему знаний.1

**Последовательность действий на Этапе 1:**

1. **Конфигурация профилей парсинга:** На основе URL-адресов, извлеченных из поля website\_urls на нулевом этапе, инженеры по данным настраивают индивидуальные записи в таблице parse\_profiles. Эти профили определяют, как часто алгоритмы должны посещать сайты организаций и какие блоки контента извлекать. Таблица sources пополняется официальными государственными реестрами и публичными базами данных.  
2. **Синхронизация с API геокодирования (Dadata):** В работу вступают скрипты, концептуально наследующие логику старого файла deepseek\_navigator.py. Используя очищенные значения ИНН или ОГРН, система делает запросы к методам dadata.find\_by\_id и dadata.suggest.1 Полученные от Dadata верифицированные адреса, а также точные значения долготы (geo\_lon) и широты (geo\_lat), записываются в таблицу venues, заменяя или дополняя сырые данные, полученные из WordPress.1  
3. **Генерация и синтез контента через LLM (Large Language Model):** Наиболее значимый шаг в улучшении качества данных. Текстовые данные, собранные краулерами со сторонних сайтов, передаются в языковую модель (на базе архитектуры DeepSeek). LLM анализирует неструктурированный текст и генерирует новые, развернутые и профессионально составленные описания организаций, полностью заменяя короткие и некачественные тексты из оригинального поля post\_content.1  
4. **Алгоритмическая классификация и строгий маппинг:** Помимо написания текстов, языковая модель анализирует спектр деятельности организации и предлагает добавить новые услуги (services) и проблемы (problem\_categories). Для обеспечения целостности данных применяется алгоритм строгой нормализации, аналогичный функции strict\_match\_array\_field из старых Python-скриптов. Вывод нейросети очищается от пунктуации (через регулярные выражения, подобные re.sub(r'\[^\\w\\s\]', '')), приводится к нижнему регистру и сверяется с жестко заданными эталонными названиями из справочников.1 Если нейросеть придумывает несуществующую услугу, система ее игнорирует, гарантируя, что в базу попадают только стандартизированные идентификаторы.  
5. **Система оценки уверенности (Confidence Scoring) и модерация:** Архитектура Navigator Core не принимает решения искусственного интеллекта вслепую. Каждому алгоритмически сгенерированному факту присваивается метрика уверенности. Если показатель высок, данные применяются автоматически со статусом «accepted». В ситуациях, когда алгоритм сомневается или источник данных противоречив, карточка организации маршрутизируется в специальный интерфейс для ручной проверки человеком-модератором.1  
6. **Непрерывное логирование аудита:** Любое добавление услуги или обновление описания приводит к автоматическому расширению поля ai\_source\_trace. Туда записывается точное время операции, URL-адрес первоисточника и версия использованного профиля из parse\_profiles, что позволяет в любой момент провести аудит качества данных.1  
7. **Формирование контентных связей:** На финальном этапе система анализирует обновленные профили организаций и автоматически генерирует связи между ними и образовательными статьями (articles). Это позволяет реализовать стратегию «Ответ вместо поиска», бесшовно направляя целевую аудиторию от информационного контента к конкретным поставщикам услуг.1

## **Архитектурные рекомендации по разработке и выполнению миграционных скриптов**

Разработка скриптов для переноса данных между столь различными архитектурами требует строгого соблюдения паттернов проектирования, характерных для экосистемы Laravel.3 Ошибки на этапе проектирования скрипта могут привести к деградации производительности и нарушению консистентности данных.

### **Применение консольных команд (Artisan Console Commands)**

Абсолютно недопустимо размещать логику переноса данных (маппинг, парсинг, ETL-преобразования) внутри стандартных файлов миграций схемы базы данных (в методах up() и down()). Структурные миграции Laravel предназначены исключительно для модификации таблиц и столбцов (DDL-запросы).4 Разработчикам следует создать специализированную консольную команду Artisan (например, php artisan navigator:legacy-import). Использование отдельного класса команды (расширяющего Illuminate\\Console\\Command) обеспечивает четкое разделение ответственности, позволяет передавать аргументы из командной строки и предоставляет удобный интерфейс для вывода прогресс-баров при длительном выполнении процесса переноса.3

### **Оптимизация потребления памяти и чанкинг (Chunking)**

Извлечение данных из WordPress-таблицы wp\_postmeta сопряжено с высоким риском возникновения проблемы N+1 запросов, когда для каждой организации совершаются десятки отдельных обращений к базе для получения каждого мета-поля. Скрипт миграции должен использовать фасад DB и Query Builder Laravel для составления единого сложного SQL-запроса, агрегирующего все метаданные в столбцы на стороне базы данных с помощью функций агрегации строк.2 Кроме того, запрещается загружать весь дамп в оперативную память сервера. Обработка данных должна производиться с использованием методов chunk() или cursor(). Пакетная обработка по 500–1000 записей за итерацию исключит вероятность сбоя по превышению лимита выделенной памяти (Memory Limit Exceeded).4

### **Управление транзакциями и временное отключение ограничений**

Этап 0 (Сырая миграция) подразумевает вставку десятков тысяч строк, включая зависимости в промежуточных таблицах (pivot). На время выполнения процесса пакетной вставки целесообразно обернуть логику в транзакцию БД DB::transaction(). Для драматического ускорения процесса записи скрипт должен программно отключить проверку ограничений внешних ключей (Foreign Key Constraints) перед началом импорта и обязательно включить их после завершения:

PHP

Schema::disableForeignKeyConstraints();  
// Логика пакетной вставки данных  
Schema::enableForeignKeyConstraints();

Помимо этого, при массовой загрузке моделей Eloquent рекомендуется временно подавить генерацию событий моделей (Model Events), чтобы избежать срабатывания лишних триггеров и наблюдателей (Observers), которые создадут нежелательную нагрузку на процессор.3

### **Идемпотентность и стратегия Upsert**

Среда выполнения миграционных скриптов не всегда стабильна. Возможны сетевые тайм-ауты, разрывы соединения с базой данных или исчерпание времени выполнения скрипта. По этой причине консольная команда миграции обязана быть строго идемпотентной. Это означает, что скрипт можно запускать неограниченное количество раз, и он не должен создавать дубликатов организаций.

Для реализации этого механизма алгоритм не должен слепо вызывать методы вставки (insert). Вместо этого, при обработке каждой сущности скрипт должен проверять наличие организации по уникальным идентификаторам (ИНН или ОГРН, аналогично старой функции find\_existing\_listing).1 В парадигме Laravel для этих целей идеально подходит метод updateOrCreate(). Если запись с указанным ИНН уже существует, скрипт лишь обновит недостающие поля; если нет — создаст новую запись. Это гарантирует абсолютную безопасность процесса миграции и возможность возобновления работы скрипта с прерванной позиции.

### **Хардкодирование строковых имен в схемах миграции**

Несмотря на то, что использование констант (например, ссылок на свойства модели Organization::TABLE\_NAME) внутри файлов структурных миграций кажется соблюдением принципа DRY (Don't Repeat Yourself), в контексте Laravel это считается антипаттерном. Модели приложения развиваются и могут менять свои константы, что приведет к фатальной поломке старых миграций при попытке развернуть проект с нуля.3 При проектировании DDL-запросов разработчики должны использовать исключительно явные, жестко заданные текстовые названия таблиц и столбцов.

Внедрение описанных архитектурных решений, строгое следование таблицам маппинга и последовательное выполнение двух этапов миграции позволят нивелировать все недостатки устаревшей EAV-структуры WordPress. Результатом станет развертывание чистой, нормализованной и интеллектуально обогащенной базы данных Navigator Core, которая послужит надежным фундаментом для построения событийной архитектуры и успешного масштабирования проекта на федеральный уровень.
