### 2. ⚠️ Исправленные ошибки в процессе выполнения

#### Ошибка 1: Проблема с префиксами таблиц WordPress

**Ошибка:**
```
SQLSTATE[42S22]: Column not found: 1054 Unknown column 'pm.meta_key' in 'field list'
```

**Причина:**
Laravel автоматически добавляет префикс `wp_` к таблицам, но при использовании алиасов в JOIN возникали конфликты.

**Решение:**
Переписан метод `chunkListings()` в `WpListingRepository` на использование raw SQL с явным указанием префиксов таблиц.

**Файл:** `app/Services/WpMigration/WpListingRepository.php`

#### Ошибка 2: Проблема с координатами PostGIS

**Ошибка:**
```
ArgumentCountError: Too few arguments to function Illuminate\Database\Query\Expression::getValue()
```

**Причина:**
Попытка использовать `DB::raw()` для координат в Eloquent `create()` не работает корректно.

**Решение:**
Изменена логика: координаты сохраняются как строка SQL в отдельном поле `_coordinates_sql`, затем обновляются через raw SQL после создания venue.

**Файл:** `app/Services/WpMigration/WpToCoreMigrator.php`

## Невыполненные задачи

### 🌐 Добавление сайтов организаций в таблицу `sources`

**Проблема:**
В процессе миграции URL из поля `hp_site` каждого листинга WordPress сохраняются в `site_urls` (JSONB массив) в таблице `organizations`, но **не создаются записи в таблице `sources`**. Это было запланировано для дальнейшего обхода сайтов организаций.

**Текущее состояние:**
- `hp_site` извлекается из WordPress через `WpListingRepository`
- Сохраняется в `organizations.site_urls` как JSONB массив через `WpToCoreMigrator::parseArrayField()`
- Записи в `sources` не создаются

**Что нужно сделать после завершения миграции:**

1. **Извлечь уникальные URL** из `organizations.site_urls`:
   ```sql
   -- Пример запроса для получения уникальных URL
   SELECT DISTINCT jsonb_array_elements_text(site_urls) as url
   FROM organizations
   WHERE site_urls IS NOT NULL AND jsonb_array_length(site_urls) > 0;
   ```

2. **Создать записи в таблице `sources`**:
   - `kind` = `'org_website'` (для сайтов организаций)
   - `base_url` = уникальный базовый URL (извлечённый из `site_urls`)
   - `name` = название организации или домен сайта
   - `entry_points` = массив точек входа (можно начать с главной страницы)
   - `is_active` = `true`
   - `last_status` = `'pending'`

3. **Рекомендации по реализации:**
   - Создать команду `php artisan navigator:import-org-websites-to-sources`
   - Извлекать базовый URL из полных URL (убрать пути, параметры)
   - Дедублицировать по `base_url` (уже есть уникальный индекс)
   - Связать с организациями через `base_url` (прямой связи нет, но можно найти через `site_urls`)

**Файлы для изменения:**
- Создать новую команду: `app/Console/Commands/ImportOrgWebsitesToSourcesCommand.php`
- Возможно, добавить метод в `WpToCoreMigrator` или создать отдельный сервис

**Примечание:** Эта задача не критична для Stage 0 миграции, но необходима для дальнейшего обхода сайтов организаций в рамках Harvester & Source Manager.

## Обнаруженные проблемы и исправления

### 🔍 Проблема 1: Дубликаты venues

**Обнаружено:**
После завершения миграции обнаружено значительное количество дубликатов venues:
- Всего venues: 13,541 (ожидалось ~6,048, по одному на организацию)
- Найдено 4,288 пар "организация + адрес" с дубликатами
- Среднее venues на организацию: 2.36 (некоторые организации имеют до 12 venues)

**Причина:**
В методе `WpToCoreMigrator::createVenue()` отсутствует проверка на существование venue с таким же адресом для данной организации. Когда несколько WP-листингов обновляют одну организацию (через `updateOrCreate` по ИНН/ОГРН), для каждого листинга создаётся новый venue, даже если адрес идентичен.

**Пример:**
Организация "Центр общения старшего поколения" получила 12 venues, из них дубликаты:
- `'г Севастополь, ул Дзержинского, д 53, ком 204/205'` — создано 3 раза
- `'г Пенза, ул Захарова, стр 20'` — создано 3 раза

**Исправление:**
Создана команда `navigator:cleanup-wp-migration-duplicates`, которая:
- Находит дубликаты venues (по `address_raw` и `organization_id`)
- Оставляет самый старый venue (первый созданный)
- Удаляет остальные дубликаты
- Удаляет связанные записи из `organization_venues`

**Результат очистки:**
- Удалено 8,033 дубликата venues
- Осталось 5,508 venues (близко к ожидаемому количеству)

**Рекомендация для будущего:**
В методе `createVenue()` добавить проверку:
```php
// Проверить, существует ли уже venue с таким адресом для этой организации
$existingVenue = Venue::whereHas('organizations', function ($q) use ($organizationId) {
    $q->where('organizations.id', $organizationId);
})
->where('address_raw', $venueData['address_raw'])
->first();

if ($existingVenue) {
    return $existingVenue; // Вернуть существующий venue
}
```

### 🔍 Проблема 2: Избыточные organizers

**Обнаружено:**
После завершения миграции обнаружено избыточное количество organizers:
- Всего organizers: 14,612 (ожидалось 6,048, по одному на организацию)
- Разница: 8,564 лишних organizers

**Причина:**
Проблема в методе `ensureOrganizer()` — проверка `if ($organization->organizer)` может не работать корректно при использовании `updateOrCreate()`, когда организация обновляется, а не создаётся заново. В результате для одной организации могло создаваться несколько organizers.

**Исправление:**
Команда `navigator:cleanup-wp-migration-duplicates`:
- Удаляет все organizers для организаций
- Создаёт ровно по одному organizer на каждую организацию

**Результат очистки:**
- Удалено 20,660 organizers (включая созданные при первой попытке очистки)
- Создано 6,048 organizers (ровно по одному на организацию)

**Рекомендация для будущего:**
В методе `ensureOrganizer()` использовать более надёжную проверку:
```php
// Использовать exists() вместо проверки отношения
if (Organizer::where('organizable_type', Organization::class)
    ->where('organizable_id', $organization->id)
    ->exists()) {
    return;
}
```
