# Архитектура миграции данных из WordPress в Navigator Core

## Обзор

Данный документ описывает архитектуру решения для миграции данных из WordPress/HivePress базы данных в Navigator Core на этапе 0 (базовая инфраструктурная миграция).

## Архитектурные принципы

### 1. Разделение ответственности

Логика миграции разделена на следующие компоненты:

- **Консольная команда** (`MigrateFromWpBaseCommand`): точка входа, управление прогрессом, обработка аргументов
- **Репозиторий WP данных** (`WpListingRepository`): извлечение и агрегация данных из WordPress EAV-структуры
- **Маппер таксономий** (`WpTaxonomyMapper`): преобразование числовых идентификаторов таксономий WP в связи с справочниками Core
- **Мигратор** (`WpToCoreMigrator`): координация процесса миграции, применение трансформаций, создание записей

### 2. Обработка EAV-структуры WordPress

WordPress использует паттерн Entity-Attribute-Value, где метаданные хранятся в таблице `wp_postmeta` в виде отдельных строк:
```
post_id | meta_key      | meta_value
--------|---------------|------------
123     | hp_inn        | 1234567890
123     | hp_ogrn       | 1234567890123
123     | hp_phone     | +7 999 123-45-67
```

**Решение**: Использование агрегирующих SQL-запросов с `GROUP_CONCAT` или `MAX(CASE WHEN ...)` для преобразования в плоскую структуру на стороне БД, избегая N+1 запросов.

### 3. Пакетная обработка (Chunking)

Для предотвращения переполнения памяти данные обрабатываются пакетами по 500-1000 записей за итерацию через метод `chunk()` или `cursor()`.

### 4. Идемпотентность

Команда использует стратегию `updateOrCreate()` по уникальным идентификаторам:
- `inn` (если присутствует)
- `ogrn` (если присутствует)
- Комбинация `inn` и `ogrn` для дедубликации

Повторный запуск команды не создает дубликатов, а обновляет существующие записи.

### 5. Трансформации данных

#### Координаты (критическая инверсия)
- `wp_postmeta.hp_latitude` → `venues.coordinates.longitude`
- `wp_postmeta.hp_longitude` → `venues.coordinates.latitude`

#### Контактные данные
- Телефоны: строка с запятыми → JSON массив (очистка от "None", пробелов)
- Email: строка с запятыми → JSON массив (валидация email)
- Сайты: строка с запятыми → JSON массив

#### Идентификаторы
- ИНН/ОГРН: очистка от нецифровых символов, валидация длины

### 6. Работа с таксономиями

Таксономии WordPress хранятся в таблицах `wp_terms` и `wp_term_relationships`. Числовые идентификаторы терминов извлекаются и маппятся на справочники Core:

- `hp_listing_category` → `thematic_categories` (через pivot `organization_thematic_categories`). Маппинг: term_id 19→7, 13→8; коды 21, 22, 23 не привязываются к thematic (они в services).
- `hp_listing_service` → развод по коду: часть → `organization_types`, часть → `specialist_profiles`, часть → `services` (через соответствующие pivot).
- `hp_listing_type` → `organization_types` (через pivot `organization_organization_types`, M:N).
- `hp_listing_ownership` → `ownership_types` (прямая связь).

**Алгоритм**: Извлечение `term_id` из связей, поиск по полю `code` в справочниках Core, создание связей через pivot-таблицы. Типы организаций и профили специалистов привязываются через pivot (несколько на организацию).

### 7. Создание системных записей

- **Sources**: Создается запись "WordPress Legacy" для маркировки происхождения данных
- **Organizers**: Создается полиморфная запись для каждой организации
- **Venues**: Создается минимум одна запись venue для каждой организации (если есть адрес/координаты)

### 8. Управление транзакциями

- Временное отключение проверки внешних ключей для ускорения массовой вставки
- Использование транзакций для обеспечения целостности данных
- Подавление событий моделей (Model Events) для снижения нагрузки

## Последовательность выполнения

1. **Инициализация**: Проверка подключения к обеим БД, создание системного источника
2. **Извлечение данных**: Пакетное чтение `hp_listing` записей из WP
3. **Трансформация**: Применение правил маппинга, очистка данных
4. **Дедубликация**: Поиск существующих организаций по ИНН/ОГРН
5. **Создание/обновление**: `updateOrCreate()` для organizations, venues, pivot-таблиц
6. **Прогресс**: Вывод прогресс-бара и статистики

## Ограничения этапа 0

- AI-поля (`ai_confidence_score`, `ai_explanation`) остаются пустыми или с дефолтами
- `works_with_elderly` устанавливается в `true` по умолчанию (требует проверки на этапе 1)
- События (events) не создаются
- Статьи (articles) переносятся только если есть соответствующие посты в WP

## Обработка ошибок

- Валидация ИНН/ОГРН: пропуск записей с некорректными идентификаторами (с логированием)
- Отсутствие координат: создание venue без координат (заполнение на этапе 1 через Dadata)
- Отсутствие таксономий: организация создается без связей (требует обогащения на этапе 1)
