# Инструкция по инициализации и рефакторингу справочников (Taxonomy Refactoring) Navigator Core

## Контекст для AI-ассистента

Мы проектируем архитектуру справочников для сервиса «Навигатор здорового долголетия» (Laravel 12). Предыдущая модель данных (мигрированная из WordPress) содержала ряд архитектурных и семантических ошибок :

1. **Потеря иерархии:** В WordPress справочник «категорий проблем» имел двухуровневую вложенность (root-узлы и дочерние элементы). При переносе он стал плоским списком.
2. **Смешение сущностей:** В справочник «Услуги» (`services`) были свалены в одну кучу реальные процессы/услуги, названия профессий врачей и типы зданий/учреждений.
3. **Негативный нейминг:** Использование слова «проблема» нарушает принципы позитивного старения (Positive Aging).

**Твоя задача:** Сгенерировать чистые DDL-миграции, Eloquent-модели и `DataSeeders` с нуля, опираясь на описанную ниже логику. Первичным ключом связности со старыми данными (для будущего импорта) является поле `code` (строка, соответствует `term_id` из WP).

## 1. Архитектурные требования и DDL-логика

Необходимо переименовать или изменить наполнение двух базовых справочников и создать новый.

### 1.1. Жизненные ситуации (`thematic_categories`)

* **Логика:** Заменяет старую таблицу `problem_categories`.
* **Иерархия:** Обязательно добавить поле `parent_id` (nullable, foreign key на `id` этой же таблицы). Это восстановит двухуровневую древовидную структуру.


### 1.2. Виды услуг (`services`)

* **Логика:** Таблица остается, но семантически очищается. Здесь лежат только *действия и процессы* (отглагольные существительные).
* **Иерархия:** Добавить `parent_id` (nullable, FK на себя) для возможности группировки услуг в будущем.


### 1.3. Типы организаций (`organization_types`)

* **Логика:** Таблица остается, но сюда мигрируют все термины-учреждения («центры», «интернаты», «отделения»), ошибочно лежавшие в старых Услугах.


* **Иерархия:** Плоский список.

### 1.4. Профили специалистов (`specialist_profiles`) — НОВЫЙ СПРАВОЧНИК

* **Логика:** Сюда переносятся все врачи, социальные работники и волонтеры из старых Услуг.
* **Иерархия:** Плоский список.
* **Связь:** Создать pivot-таблицу `organization_specialist_profile` (`organization_id`, `specialist_profile_id`), возможно – связь с Articles (дай аргументы за/против)

## 3. Инструкции по реализации для Cursor

1. **Создай/Измени DDL-миграции:** Напиши миграции для создания таблиц строго в следующем порядке (с учетом внешних ключей): `thematic_categories`, `services`, `organization_types`, `specialist_profiles`.
2. **Создай Pivot-таблицы:** Сгенерируй миграции для связей.
3. **Напиши Seeder:** Создай `Database\Seeders\TaxonomySeeder.php` (или аналог). Сложи в него массивы из этой инструкции и используй цикл с `updateOrCreate(['code' => $item['code']], $item)`, чтобы скрипт был идемпотентным.
4. **Разрешение связей (Родители):** В сидере для `thematic_categories` сначала создай записи с `parent_code = null`, а затем вторым проходом создай дочерние элементы, находя `id` родителя по его `code`.
