# Ветки и релизы: develop vs main, миграции и очистка

Кратко: как работать в ветках, чтобы в dev сохранить все миграции и временные команды, а в test/prod отдавать чистую версию без рудиментов WP.

---

## 1. Ветки: develop и main

**Рекомендация:** не работать по умолчанию в `main`.

- **`main`** — только «готово к тесту/проду». Сюда мержим только проверенный код, без отладочных команд и без обязательных следов миграции из WP.
- **`develop`** (или `dev`) — основная рабочая ветка. Все фичи, миграции-патчи, промежуточные команды, эксперименты живут здесь.

### Первоначальная настройка (один раз)

```bash
# Убедиться, что main в актуальном состоянии
git checkout main
git pull

# Создать ветку develop от main и переключиться на неё
git checkout -b develop

# Пушить и при желании задать upstream
git push -u origin develop
```

Дальше по умолчанию работаете в `develop`, коммитите и пушите в `origin develop`. В `main` попадаете только через мерж при подготовке релиза.

### Подготовка к тесту/проду (релиз)

1. Убедиться, что в `develop` всё закоммичено, тесты зелёные.
2. Выполнить шаги очистки (см. ниже): удалить рудименты WP, при необходимости обновить миграции (schema dump).
3. Закоммитить очистку в `develop`.
4. Смержить `develop` в `main`:
   ```bash
   git checkout main
   git merge develop --no-ff -m "Release: описание"
   git push origin main
   ```
5. Тест/прод деплоит из `main` (или из тега с `main`).

При необходимости можно ввести ветку `release/YYYY-MM-DD`: ответвиться от `develop`, на ней сделать финальную очистку и проверки, затем мерж в `main` и в `develop`.

## 2. Миграции: не удалять, а «подменить» для новых окружений

Уже выполненные миграции **не удаляйте** из репозитория: в существующих БД (в т.ч. у вас в dev) в таблице `migrations` записаны их имена. Удаление файлов приведёт к ошибкам при следующем `php artisan migrate`.

Чтобы при первом деплое на **новое** окружение (test/prod) не гонять десятки миграций по порядку, используйте **schema dump** Laravel:

```bash
# В dev, после того как все миграции применены и схема стабильна
cd backend
php artisan schema:dump
```

Laravel создаст файл в `database/schema/` (например, `mysql-schema.sql` или для pgsql свой). Этот файл коммитится в репозиторий. При первом запуске `php artisan migrate` на **пустой** БД Laravel сначала загрузит этот дамп, затем выполнит только те миграции, которые **новее** дампа. В итоге новое окружение получает актуальную схему одним шагом плюс только последние миграции.

- Старые файлы миграций **остаются** в репо (история и совместимость с уже существующими БД).
- Опция `--prune` у `schema:dump` **удаляет** старые миграции — используйте её только если уверены, что все окружения будут подниматься с нуля по дампу (иначе сломаете уже запущенные инстансы).

Итого: в dev можно спокойно накапливать миграции-патчи; перед релизом обновляете schema dump; при переносе в тест/прод новые инстансы получают «объединённую» схему через дамп.

## 3. Очистка рудиментов миграции из WP

Проводится **в develop** (или в ветке release) перед мержем в `main`. Пошагово — по [wp_migration_cleanup_guide.md](./wp_migration_cleanup_guide.md).

Кратко что убрать к моменту теста/прода:

- Подключение `mysql_wp` в `config/database.php`
- Директория `app/Services/WpMigration/`
- Команды: `MigrateFromWpBaseCommand`, `CleanupWpMigrationDataCommand`, `CleanupWpMigrationDuplicatesCommand`, `ManualMigrateListings7419And4425` (и другие одноразовые ручные команды миграции)
- Из `.env.example` — переменные `DB_WP_*`

Промежуточные команды, полезные только в dev (например, отчёты, проверки консистентности), можно либо оставить в коде (они не трогают WP), либо вынести в отдельные ветки/скрипты — по желанию.

## 4. Краткий чек-лист перед мержем develop → main

- [ ] В `develop` выполнена очистка по [wp_migration_cleanup_guide.md](./wp_migration_cleanup_guide.md)
- [ ] Обновлён schema dump: `php artisan schema:dump` (миграции применены), файл в `database/schema/` закоммичен
- [ ] Тесты проходят, линтеры без ошибок
- [ ] В `main` смержен `develop`, тег релиза поставлен при необходимости

После этого деплой теста/прода из `main` будет без следов WP и с аккуратной историей миграций для новых БД.

## 5. После мержа main → develop: дубликаты от IDE/мержа

При мерже (или из-за IDE) иногда появляются дубликаты: файлы/папки с суффиксом ` 2` (пробел + «2»), например `Commands 2/`, `file 2.php`. Они не должны попадать в репозиторий.

- В `backend/.gitignore` и `ai-pipeline/harvester/.gitignore` уже есть правила `* 2.*`, `* 2`, `**/* 2/` — такие пути не коммитятся.
- После мержа проверьте и удалите их с диска:  
  `find . -type d -name '* 2*' ; find . -type f -name '* 2*'`  
  затем `rm -rf "path/to/Commands 2"` и т.п.
- Если такие файлы/папки уже попали в коммит — удалите их через `git rm` и закоммитьте удаление.
