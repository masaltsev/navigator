# Обогащение адресов площадок (venues) через DaData

## Зачем это нужно

После миграции из WordPress у большинства площадок заполнен только **сырой адрес** (`address_raw`). Поля **`fias_id`** и **`kladr_id`** в миграторе явно оставлены пустыми с комментарием «Will be filled in Stage 1 via Dadata». Из‑за этого:

1. **Не работает фильтр API по городу**  
   Эндпоинт `GET /api/v1/organizations` поддерживает параметр **`city_fias_id`**: выборка идёт по площадкам, у которых `venues.fias_id LIKE :city_fias_id%`. Если у площадок `fias_id` пустой, фильтр не находит ни одной организации. Приходится искать «Вологда» и другие города вручную по вхождению подстроки в `address_raw` (и только на стороне клиента или в своих скриптах).

2. **Нет единого ключа для связки с ФИАС/КЛАДР**  
   Без `fias_id` нельзя надёжно сопоставлять площадки с внешними справочниками, геокодировать и фильтровать по иерархии (регион → город → улица).

3. **Координаты и регион не унифицированы**  
   В WP миграции координаты и `region_iso` берутся из полей листинга; при ошибках или пустых значениях часть площадок остаётся без координат и без кода региона. DaData может дополнить и уточнить и координаты, и регион.

---

## Разделение ролей: fias_id vs coordinates

В нашей логике:

- **`fias_id`** у площадки — уровень **населённого пункта** (город, село и т.п.). Нужен для фильтра «в каком городе»: API-параметр `city_fias_id` отбирает организации, у которых у площадки `fias_id` совпадает с переданным (или начинается с него). Точный адрес (улица, дом) для фильтрации не используем — только «привязка к городу».
- **Точное гео и радиус** — по полю **`coordinates`** (PostGIS Point). Фильтр `lat`, `lng`, `radius_km` в API считает расстояние от точки до площадки и отбирает по `ST_DWithin(coordinates, ...)`.

Итого: fias_id = населённый пункт (для фильтра по городу), координаты = точное место и поиск «рядом».

## Текущая структура таблицы `venues`

| Колонка        | Тип              | Заполненность после WP-миграции | Назначение |
|----------------|------------------|----------------------------------|------------|
| `id`           | uuid             | да                              | PK         |
| `address_raw`  | string           | да (из `hp_location`)           | Исходный адрес для обогащения |
| **`fias_id`**  | string, nullable | **почти всегда NULL**          | ФИАС **населённого пункта** (город/село); для фильтра `city_fias_id` в API |
| **`kladr_id`** | string, nullable | **NULL**                        | Код КЛАДР (при необходимости) |
| **`region_iso`** | string, nullable | частично (из `hp_region`)       | Код региона (например RU-VLG); можно уточнить через DaData |
| `coordinates`  | geometry(Point, 4326) | частично (из WP lat/lng)   | **Точное гео**; поиск по радиусу в API идёт по этой точке |
| timestamps, soft deletes | —            | да                              | —          |

В коде миграции из WP явно задано: `fias_id => null`, `kladr_id => null`; `region_iso` заполняется из `hp_region` через `normalizeRegion()`.

## Как API использует адреса и `fias_id`

- **Список организаций**  
  В выборку попадают организации, у которых есть площадка с подходящим `fias_id`:

  ```php
  // OrganizationController: фильтр по городу
  if ($request->filled('city_fias_id')) {
      $q->whereHas('venues', fn ($q) =>
          $q->where('fias_id', 'LIKE', $request->input('city_fias_id') . '%')
      );
  }
  ```

  Передаётся **FIAS-код населённого пункта** (город и т.п.). В выборку попадают площадки, у которых в `fias_id` хранится тот же НП (совпадение или префикс). У нас в `venue.fias_id` храним именно уровень населённого пункта. Если у площадок `fias_id` не заполнен, фильтр по городу не работает.

- **Поиск «Вологда» и т.п.**  
  Сейчас это можно сделать только по тексту в `address_raw` (например, подстрока «Вологда»), причём не в API, а у себя в скриптах или на клиенте. После проставления `fias_id` через DaData станет возможна нормальная фильтрация по `city_fias_id` (передав FIAS код Вологды).

## Что должен делать скрипт обогащения через DaData

1. **Источник данных**  
   Обходить все (или выбранные) записи в `venues`, у которых есть непустой `address_raw`.

2. **Вызов DaData**  
   Для каждого такого адреса вызывать API стандартизации/подсказок DaData, чтобы получить:
   - **fias_id** — код ФИАС **населённого пункта** (город, село). В нашей модели в `venue.fias_id` храним именно уровень НП, а не дома/улицы; при обогащении нужно брать из ответа DaData соответствующий уровень (например, из структуры с `data.region_fias_id` / городом или аналогом в выбранном методе API).
   - при необходимости **kladr_id**;
   - при необходимости **region_iso** (или код региона в том виде, в котором вы храните в `region_iso`);
   - **координаты** (geo_lat, geo_lon) — точное гео; поиск по радиусу в API идёт по ним. Если у площадки координат ещё нет — подставить из DaData.

3. **Обновление БД**  
   Записывать в `venues`:
   - `fias_id` — обязательно для работы фильтра по городу;
   - по желанию: `kladr_id`, `region_iso`, `coordinates` (если решите обновлять координаты из DaData).

4. **Ограничения и надёжность**  
   - Учитывать лимиты и rate limit API DaData (паузы, батчи, очередь).
   - Решить: обновлять только записи с пустым `fias_id` или перезаписывать все (например, с флагом `--force`).
   - Логировать ошибки и неуспешные ответы (не найдено, таймаут и т.д.), при необходимости писать отчёт по площадкам без `fias_id` после прогона.
   - Имеет смысл предусмотреть **dry-run** (без записи в БД) и выборочный прогон (например, только определённые регионы или id площадок).

---

## Рекомендуемые шаги при реализации

1. Получить API-ключ DaData и выбрать метод (подсказки по адресу / стандартизация — в зависимости от того, нужны ли координаты и один «лучший» вариант).
2. Выбрать стратегию обновления: только `fias_id` или также `kladr_id`, `region_iso`, `coordinates`.
3. Реализовать цикл по `venues` с непустым `address_raw`, вызов DaData, маппинг ответа в поля таблицы.
4. Добавить ограничение частоты запросов и повтор при временных ошибках.
5. После прогона проверить фильтр `GET /api/v1/organizations?city_fias_id=<fias_вологды>` — должны появиться организации с площадками в Вологде (без ручного поиска по «Вологда» в адресе).

После этого те же карточки, что вы собирали вручную по подстроке «Вологда» в адресе, можно будет получать через API с параметром `city_fias_id`.
